# План по артефактам вывода инструментов и раздуванию контекста

Дата: 24.04.2026

## Проблема

Сессии Hermes Agent чрезвычайно быстро заполняют контекст при обычных рабочих процессах с интенсивным использованием инструментов. В одной из сессий Workspace (`cda915a9-fbf6-4bbf-9617-e7e9d26f40be`):

- Всего 256 сообщений
- 7 сообщений пользователя
- 117 сообщений ассистента
- 132 сообщения инструментов
- ~603 тыс. символов вывода инструментов
- ~152 тыс. приблизительных токенов

Раздувание происходит не из-за диалога пользователя и ассистента. В основном это сохраненный след выполнения (execution trace):

- большие фрагменты `read_file`
- логи терминала/тестов
- полная документация `skill_view`
- списки issue/PR на GitHub
- выводы патчей/диффов

Это приводит к следующим последствиям:

1. Окно контекста заполняется быстро, несмотря на обычное поведение Claude.
2. Предупреждения об автоматическом сжатии (compaction) появляются слишком рано и часто.
3. В интерфейсе чата Workspace начинают доминировать карточки инструментов, и кажется, что сообщения пользователя и ассистента исчезли.
4. Модель принудительно получает огромные выводы инструментов на каждом шаге, даже когда ей нужны только ссылки или сводки.

## Ключевая идея

В настоящее время Claude смешивает две разные концепции:

1. **Контекст разговора/модели** — то, что нужно LLM для рассуждений.
2. **След выполнения/артефакты** — полные выводы инструментов, логи, содержимое файлов, диффы, документация навыков.

Их следует разделить.

## Желаемая архитектура

Использование **выводов инструментов на базе артефактов + компактная проекция контекста**.

### Транскрипт чата должен содержать:

- сообщения пользователя
- финальные сообщения ассистента
- компактные сводки/ссылки на инструменты

Пример компактного сообщения инструмента:

```json
{
  "role": "tool",
  "content": {
    "tool": "read_file",
    "summary": "Прочитаны строки 680-1499 файла src/screens/chat/components/chat-message-list.tsx",
    "chars": 36334,
    "artifact_id": "toolout_abc123",
    "truncated": true
  }
}
```

### Артефакты Инспектора должны содержать:

Полные выводы инструментов, логи, артефакты, загружаемые лениво:

```ts
type InspectorArtifact = {
  id: string
  sessionId: string
  toolCallId?: string
  kind: 'tool_output' | 'file_read' | 'terminal_log' | 'diff' | 'skill_doc'
  title: string
  summary: string
  preview: string
  contentSize: number
  contentPath?: string
  createdAt: number
}
```

### Контекст модели должен получать:

По умолчанию только компактные сводки:

```txt
Инструмент read_file завершил работу.
Сводка: Прочитаны строки 680-1499 файла chat-message-list.tsx.
Полный вывод сохранен как артефакт toolout_abc123.
Используйте artifact_read(toolout_abc123, offset, limit) при необходимости.
```

## Текущее состояние Инспектора Workspace

Проверенный код:

- `src/components/inspector/activity-store.ts`
  - Только хранилище Zustand в памяти
  - сохраняет `{ type, time, text }`
  - не является долговечным; данные пропадают при обновлении страницы
- `src/components/inspector/inspector-panel.tsx`
  - Вкладка Artifacts фильтрует `events.filter(e => e.type === 'artifact')`
  - отображает только текст и время
  - нет полного содержимого, ID, предпросмотра или ленивой загрузки

Таким образом, правая панель — это правильное место для UX, но ей требуется долговечное фоновое хранилище.

## Варианты реализации

### Вариант A — Каноническое хранилище артефактов Hermes Agent

Лучший долгосрочный вариант, так как от него выигрывают все клиенты: CLI, шлюз, Workspace, будущий WebUI.

Добавить таблицу/хранилище в Hermes Agent:

```sql
tool_artifacts (
  id TEXT PRIMARY KEY,
  session_id TEXT,
  message_id INTEGER,
  tool_name TEXT,
  kind TEXT,
  title TEXT,
  content_path TEXT,
  content_preview TEXT,
  content_size INTEGER,
  created_at REAL
)
```

Большое содержимое на диске:

```txt
~/.hermes/sessions/artifacts/<session_id>/<artifact_id>.json
```

Эндпоинты API:

```txt
GET /api/sessions/:sessionId/artifacts
GET /api/artifacts/:artifactId
```

### Вариант B — MVP-кэш артефактов в Workspace

Более быстрый патч только для Workspace.

Хранить артефакты в:

```txt
~/hermes-workspace/.runtime/artifacts/<session_id>/<artifact_id>.json
```

или в небольшой локальной БД / JSON-индексе.

Workspace в `/api/history` или `/api/send-stream` выносит огромные результаты инструментов наружу перед отрисовкой или отправкой в React.

Хороший MVP, но недостаточно для борьбы с раздуванием контекста на уровне модели, если только построитель контекста в Hermes Agent также не перестанет повторно внедрять полную полезную нагрузку инструмента.

## Предложение по политике

Использовать порог размера:

```ts
const INLINE_TOOL_OUTPUT_LIMIT = 4_000
```

Если вывод инструмента <= 4 тыс. символов:

- оставлять встроенным (inline)

Если > 4 тыс. символов:

- сохранять полный вывод как артефакт
- заменять содержимое чата/сессии сводкой и ссылкой на артефакт
- показывать компактную карточку в чате
- полное содержимое доступно в Инспекторе

## Настройки для конкретных инструментов

| Тип инструмента | Политика контекста |
|---|---|
| `read_file` | Сохранять полностью как артефакт; контекст получает путь к файлу, диапазон строк, предпросмотр/фрагменты |
| `search_files` | Оставлять компактные совпадения встроенными |
| Успех `terminal` | Сохранять полный лог; контекст получает команду, код выхода, последние ~20 строк |
| Ошибка `terminal` | Сохранять полный лог; контекст получает команду, код выхода, соответствующую ошибку/хвост лога |
| `skill_view` | Сохранять/ссылаться на документацию навыка; контекст получает название/версию/сводку/хэш навыка |
| `browser_snapshot` | Сохранять полный снимок/артефакт; контекст получает сводку страницы |
| `patch` | Оставлять сводку/полный дифф, если он небольшой; выносить большие диффы вовне |
| `todo` | Только компактное состояние |
| Списки issue/PR GitHub | Сохранять полный список; контекст получает количество и первые N записей |

## Текущий диагноз: Workspace vs Hermes Agent

Завершающая карточка "Работа инструментов завершена" — это защитный механизм Workspace, а не желаемое стабильное состояние.

Вероятное разделение ответственности:

1. **Первоисточником является сохранение сессии в Hermes Agent**, когда сохраненная история заканчивается сообщениями инструментов без финального текста ассистента. Здоровый транскрипт агента должен завершать каждый шаг использования инструментов одним видимым для пользователя результатом ассистента: финальным ответом, ошибкой, прерыванием/отменой, отсутствием действий или компактной сводкой работы инструментов.
2. **В Workspace были ошибки UI/отрисовки**, которые усугубляли проблему:
   - серверная роль `role: "tool"` не нормализовалась во фронтенд-роль `toolResult`
   - `toolCallId` / `toolName` не поднимались наверх последовательно
   - завершающие строки только с инструментами могли визуально доминировать в чате
   - гигантские выводы инструментов отрисовывались встроенными вместо использования артефактов
3. **Наша текущая настройка/сессия усиливает это**, так как данный диалог чрезвычайно насыщен инструментами и близок к максимуму контекста; действия разработчика/инструмента после понятного человеку ответа ассистента могут оставить в конце сохраненные строки инструментов.

Вывод: Workspace должен превентивно сворачивать такие состояния или показывать сводку инструментов, но каноническое исправление должно быть в коде записи потока/сессии Hermes Agent: никогда не сохранять завершенный шаг чата, который заканчивается только выводом инструментов.

## Рекомендуемые следующие шаги

1. Оставить защитный механизм Workspace, но рассматривать его как обработку пограничных состояний.
2. Продолжить каноническую работу над артефактами в Hermes Agent: выносить вовне слишком большие выводы инструментов до того, как они попадут в сохраненный контекст или контекст модели, а не только перед отрисовкой истории в Workspace.
3. Продолжить полировку Workspace:
   - явное действие "Открыть артефакт" на компактных карточках результатов инструментов
   - прямая ссылка на вкладку Artifacts в Инспекторе
   - политика очистки/хранения артефактов

## Ключевой принцип

Полный вывод инструментов должен находиться в артефактах/логах; контекст модели должен получать только сводки, ссылки и лениво загружаемые фрагменты.
