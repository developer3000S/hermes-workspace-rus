# Пресеты ролей Swarm

Диалог Add Swarm поставляется с пресетами ролей, чтобы новые агенты Hermes могли начинать работу с реальным операционным контрактом, а не с пустого текстового поля и оптимизма. Выберите наиболее подходящий пресет, настройте миссию, а затем запустите воркера.

У каждой роли есть:

- специализация
- навыки по умолчанию
- модель по умолчанию
- когда ее использовать
- ссылка на каноническую спецификацию

Канонические спецификации проектов находятся в директории спецификаций swarm по пути:

```text
/swarm-specs/projects/<worker>.md
```

Используйте эти спецификации как источник истины для постоянных миссий. Пресет роли — это форма для быстрого старта; спецификация проекта — это долгосрочный контракт.

## Сводка пресетов

| Пресет | Модель по умолчанию | Когда использовать |
| --- | --- | --- |
| Orchestrator | GPT-5.4 | Вам нужны диспетчеризация, маршрутизация, обнаружение отклонений и эскалация. |
| Builder | GPT-5.5 | Вам нужен продуктовый код, поставляемый с доказательствами тестов/сборки. |
| Reviewer | GPT-5.4 | Вам нужна проверка с подтверждением байтов и готовность к слиянию. |
| Triage | GPT-5.5 | Вам нужны оцененные, воспроизведенные, исправленные или подготовленные issues/PR. |
| Lab | GPT-5.4 | Вам нужны изолированные эксперименты или бенчмаркинг локальных моделей. |
| Sage | GPT-5.5 | Вам нужны исследования, синтез, скрипты или тексты для запуска. |
| Scribe | GPT-5.5 | Вам нужны доки, спецификации, передачи, гигиена навыков, курирование памяти. |
| Foundation | GPT-5.4 | Вам нужна работа с рантаймом, ремонтом, инфраструктурой, исправностью или жизненным циклом. |
| QA | GPT-5.4 | Вам нужны регрессионные, дымовые тесты и верификация "ожидаемое против фактического". |
| Mirror Integrations | GPT-5.4 | Вам нужны синхронизация с апстримом, интеграции или наборы ресурсов (assets). |
| Custom | на выбор пользователя | Вы создаете канал, который не вписывается в существующий пресет. |

## Orchestrator (Оркестратор)

Специализация: состояние плоскости управления, диспетчеризация, обнаружение отклонений, эскалация.

Навыки по умолчанию:

- `swarm-orchestrator`
- `swarm-worker-core`
- `swarm-review-learning-loop`
- `self-improvement`

Модель по умолчанию: GPT-5.4

Когда использовать:

- маршрутизация миссий с участием нескольких воркеров
- перевод намерений в SwarmBriefs
- интерпретация контрольных точек
- обнаружение отклонений
- повторные промпты воркерам
- эскалация блокировщиков
- управление постоянными миссиями

Каноническая спецификация:

```text
/swarm-specs/projects/swarm3.md
```

Хорошая контрольная точка:

```text
STATE: HANDOFF
RESULT: Routed docs release to swarm7 and review gate to swarm6 after docs checkpoint.
NEXT_ACTION: Wait for swarm7 NEEDS_REVIEW checkpoint, then dispatch swarm6 byte-verified review.
```

## Builder (Строитель)

Специализация: full-stack реализация, быстрые циклы поставки.

Навыки по умолчанию:

- `swarm-worker-core`
- `byte-verified-code-review`

Модель по умолчанию: GPT-5.5

Когда использовать:

- UI продукта
- эндпоинты бэкенда
- интеграции
- исправление багов
- функциональные срезы
- сфокусированные рефакторинги

Примеры канонических спецификаций:

```text
/swarm-specs/projects/swarm5.md
/swarm-specs/projects/swarm10.md
```

Builder должен поставлять узкие диффы, а не перестраивать весь собор только потому, что кнопка выглядела одиноко.

## Reviewer (Рецензент)

Специализация: проверка кода с подтверждением байтов, именование, тесты, шлюз сборки.

Навыки по умолчанию:

- `swarm-worker-core`
- `byte-verified-code-review`
- `swarm-review-learning-loop`

Модель по умолчанию: GPT-5.4

Когда использовать:

- готовность PR
- шлюзы релизных веток
- проверка корректности сгенерированных файлов
- хрупкие изменения именований
- проверка регрессий
- верификация сборки/тестов

Каноническая спецификация:

```text
/swarm-specs/projects/swarm6.md
```

Вердикты рецензента должны быть одними из:

- APPROVED (ОДОБРЕНО)
- CHANGES_REQUESTED (ТРЕБУЮТСЯ ИЗМЕНЕНИЯ)
- BLOCKED (ЗАБЛОКИРОВАНО)

## Triage (Триаж)

Специализация: автономная обработка PR/issues.

Навыки по умолчанию:

- `swarm-worker-core`
- `byte-verified-code-review`
- `swarm-review-learning-loop`

Модель по умолчанию: GPT-5.5

Когда использовать:

- сканирование бэклога issues
- триаж обратной связи по PR
- заметки о воспроизведении
- минимальные провальные тесты
- ветки с небольшими исправлениями
- ранжирование issues

Примеры канонических спецификаций:

```text
/swarm-specs/projects/swarm12.md
/swarm-specs/projects/swarm1.md
```

Triage никогда не должен молча сливать или закрывать задачи. Он подготавливает работу и запрашивает проход через шлюз.

## Lab (Лаборатория)

Специализация: R&D локальных моделей, spec-dec, бенчмаркинг.

Навыки по умолчанию:

- `swarm-worker-core`
- `pc1-ollama-gguf-bench`
- `swarm-bench-worker`

Модель по умолчанию: GPT-5.4

Когда использовать:

- тестирование локальных моделей
- сравнение пропускной способности
- спекулятивные улучшения рантайма
- изолированные прототипы
- логи экспериментов

Каноническая спецификация:

```text
/swarm-specs/projects/swarm9.md
```

Lab позволено быть странной, потому что она изолирована. Продуктовым каналам — нет.

## Sage (Мудрец)

Специализация: исследования, скрипты, контент для X, творческие брифы.

Навыки по умолчанию:

- `swarm-worker-core`
- `last30days`
- `pdf-and-paper-deep-reading`

Модель по умолчанию: GPT-5.5

Когда использовать:

- технические исследования
- сканирование рынка/моделей
- ракурсы для запуска
- черновики тредов
- творческие брифы
- цитирование

Каноническая спецификация:

```text
/swarm-specs/projects/swarm4.md
```

Sage пишет черновики; люди одобряют публичную публикацию.

## Scribe (Писарь)

Специализация: доки, гигиена навыков, курирование памяти.

Навыки по умолчанию:

- `swarm-worker-core`
- `last30days`
- `creative-writing`

Модель по умолчанию: GPT-5.5

Когда использовать:

- обновления README
- деревья документации
- примечания к релизу
- передачи (handoffs)
- спецификации
- руководства (runbooks)
- документация навыков
- отчеты о гигиене памяти

Каноническая спецификация:

```text
/swarm-specs/projects/swarm7.md
```

Scribe делает систему понятной. Это менее гламурно, чем создание системы, и обычно именно это определяет, сможет ли кто-то ею пользоваться.

## Foundation (Фундамент)

Специализация: инфраструктура, плейбук ремонта, настройка автопилота.

Навыки по умолчанию:

- `swarm-worker-core`

Модель по умолчанию: GPT-5.4

Когда использовать:

- API рантайма
- проверки исправности (health checks)
- жизненный цикл tmux
- обнаружение профилей
- обновления плейбука ремонта
- настройка цикла оркестратора
- контракты бэкенда

Примеры канонических спецификаций:

```text
/swarm-specs/projects/swarm8.md
/swarm-specs/projects/swarm2.md
```

Foundation не дает полу под ногами превратиться в суп.

## QA (Обеспечение качества)

Специализация: регрессионное QA, верификация рендеринга, проверки "ожидаемое против фактического".

Навыки по умолчанию:

- `swarm-worker-core`
- `byte-verified-code-review`

Модель по умолчанию: GPT-5.4

Когда использовать:

- дымовые тесты (smoke tests)
- проверки регрессий
- проходы "ожидаемое против фактического" в UI
- верификация артефактов
- уверенность после сборки
- исправность релиза

Каноническая спецификация:

```text
/swarm-specs/projects/swarm11.md
```

QA должен точно сообщать, что было проверено, что именно не удалось и как именно это воспроизвести.

## Mirror Integrations (Интеграции Mirror)

Специализация: наборы ресурсов, синхронизация с апстримом, интеграции.

Навыки по умолчанию:

- `swarm-worker-core`
- `claude-promo`
- `songwriting-and-ai-music`

Модель по умолчанию: GPT-5.4

Когда использовать:

- отслеживание диффов апстрима
- упаковка интеграций
- сбор ресурсов (assets)
- генерация творческих ресурсов
- поддержка кросс-каналов

Каноническая спецификация:

```text
/swarm-specs/projects/swarm10.md
```

Mirror Integrations — это канал для портативных полезных вещей, а не для случайных блестящих отвлекающих факторов. Но случайные блестящие факторы будут появляться. Они хитрые.

## Custom (Пользовательская)

Специализация: определяется пользователем.

Навыки по умолчанию: нет.

Модель по умолчанию: на выбор пользователя.

Когда использовать:

- воркер не вписывается в существующую роль
- вы тестируете новый канал
- вам нужна временная специализация
- прототипируется будущий пресет

Каноническая спецификация:

```text
/swarm-specs/projects/<new-worker>.md
```

Пользовательские воркеры все равно должны получить:

- `swarm-worker-core`
- описание роли
- специализацию
- миссию
- контракт контрольной точки
- четкую границу одобрения

Пустые пользовательские воркеры становятся дорогим автодополнением. Сначала добавьте структуру.

## Выбор правильной роли

Используйте это правило маршрутизации:

| Если работа в основном заключается в... | Отправьте ее... |
| --- | --- |
| решении, кто и что должен делать | Orchestrator |
| изменении кода продукта | Builder |
| доказательстве безопасности ветки | Reviewer |
| разборе issues/PR | Triage |
| экспериментах вне релиза | Lab |
| исследованиях или написании текстов | Sage |
| объяснении, документировании, сохранении контекста | Scribe |
| инфраструктуре рантайма/исправности/ремонта | Foundation |
| проверке поведения и регрессий | QA |
| апстриме/интеграциях/ресурсах | Mirror Integrations |
| ничем из вышеперечисленного | Custom |

## Добавление нового пресета роли

1. Определите роль в списке пресетов UI.
2. Дайте ей специализацию в одну строку.
3. Дайте ей постоянную миссию.
4. Выберите навыки по умолчанию.
5. Выберите модель по умолчанию.
6. Создайте каноническую спецификацию проекта.
7. Добавьте ее в этот документ.
8. Отправьте крошечную дымовую задачу и проверьте контрольную точку.

Если шаг 6 кажется слишком трудоемким, роль, скорее всего, еще не является реальной.
