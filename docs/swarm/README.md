# Режим Swarm (Рой)

Режим Swarm — это плоскость управления Hermes Workspace для агентов Hermes: постоянные воркеры, работающий оркестратор, шлюз проверки и достаточная видимость рантайма, чтобы система была понятной, а не мистической.

Обещание этого релиза простое:

- Может существовать неограниченное количество агентов Hermes.
- Один оркестратор переводит намерения в диспетчеризацию.
- Людям не нужно вручную маршрутизировать каждую задачу.
- У каждого воркера есть роль, профиль, миссия и контракт контрольной точки.
- Каждое рискованное действие по-прежнему проходит через Greenlight Gate (Шлюз одобрения).

Это не просто чат-оболочка с вкладками. Это операционная поверхность для локального роя агентов.

## С чего начать

- [QUICKSTART.md](./QUICKSTART.md) — клонируйте, запустите, определите профили, создайте воркеров, отправьте первую задачу.
- [ARCHITECTURE.md](./ARCHITECTURE.md) — цикл, структура SwarmBrief, маршрутизация уведомлений, каналы, проверка, ремонт.
- [SKILLS.md](./SKILLS.md) — встроенные навыки роя, автозагрузка и правила создания пользовательских навыков.
- [ROLES.md](./ROLES.md) — пресеты ролей, используемые в диалоге Add Swarm и канонических спецификациях проекта.

## Модель "за 30 секунд"

Eric говорит с Aurora. Aurora превращает намерение в бриф. Оркестратор направляет этот бриф нужному агенту Hermes. Воркеры выполняют задачи внутри постоянных сессий tmux, ставят контрольные точки с доказательствами, а оркестратор решает, что делать дальше: продолжить, исправить, эскалировать или поместить карточку во Входящие (Inbox).

```text
Eric -> Aurora -> swarm3/оркестратор -> воркеры ролей -> контрольные точки -> отчеты/inbox -> проверка/эскалация
```

Важный сдвиг в том, что диспетчеризация становится системой, а не просто "атмосферой". Воркер — это не просто "еще один вызов модели". Это именованный канал с памятью, состоянием рантайма, набором навыков, профилем и работой.

## Что входит в v1

### Чат оркестратора

Чат оркестратора — это основная поверхность управления. Используйте его, чтобы запросить одно действие, декомпозированный план или широковещательную рассылку. Он может маршрутизировать задачи конкретным воркерам, создавать миссии, ждать контрольных точек и отправлять уточняющие промпты, если воркер отклонился от цели.

### Плоскость управления мультиагентами

Поверхность Swarm показывает воркеров в виде операционных карточек: роль, состояние, текущая задача, модель, недавний сигнал, членство в комнате и доступные действия. Вы можете изучать топологию вместо того, чтобы гадать, какой агент жив.

### Канбан-доска задач

TaskBoard дает рою поверхность для планирования: бэклог, готовность, выполнение, проверка, заблокировано, готово. Она намеренно сделана скучной. Скучное состояние задач лучше, чем красивое кладбище полузаброшенных чатов.

### Отчеты + Входящие (Reports + Inbox)

Отчеты и Входящие — это места, где работа роя становится доступной для проверки. Контрольные точки с пометкой `NEEDS_REVIEW`, блокировщики, передачи и заслуживающие эскалации сводки попадают сюда, чтобы Eric мог одобрить те немногие вещи, которые требуют оценки.

### Встроенный TUI View

Представление рантайма подключается к воркерам на базе tmux, когда они доступны. Если tmux недоступен, рабочее пространство переключается на оболочку (shell) или чтение логов. Цель — прямая наблюдаемость: если агент Hermes что-то делает, вы можете видеть его канал.

## Основные термины

| Термин | Значение |
| --- | --- |
| Агент Hermes | Именованный постоянный воркер с ролью, профилем, навыками и состоянием рантайма. |
| Оркестратор | Агент Hermes, ответственный за диспетчеризацию, обнаружение отклонений, маршрутизацию и эскалацию. |
| SwarmBrief | Каноническая форма задачи, отправляемая от оркестратора воркеру. |
| Постоянная миссия | Бессрочная обязанность, к которой воркер возвращается в режиме простоя. |
| Разовое задание | Однократная задача, отправляемая через тот же контракт контрольной точки. |
| Контрольная точка | Блок статуса с доказательствами, возвращаемый воркером. |
| Greenlight Gate | Граница одобрения человеком для необратимых или внешне видимых действий. |
| Плейбук ремонта | Известные сбои, сопоставленные с безопасными исправлениями перед эскалацией. |

## Ментальная модель для пользователей

Рабочее пространство дает вам три уровня контроля:

1. Запросите результат у оркестратора.
2. Проверяйте и направляйте миссию из плоскости управления.
3. Погружайтесь в рантайм воркера только тогда, когда вам нужны точные доказательства.

Вам не нужно нянчиться с каждым шагом. Вы должны иметь возможность запросить проверку документации релиза, увидеть, как воркер по документации берет задачу, наблюдать за появлением контрольной точки, отправить следом канал проверки и одобрить PR только тогда, когда проверка подтвердит его готовность.

## В чем хорош режим Swarm

- Конвейеры релизов с этапами документации, сборки, проверки, QA и PR.
- Автономный триаж issues с ограниченными каналами ремонта.
- Циклы "исследование + сборка", где один воркер проводит разведку, а другой поставляет результат.
- Длительные лабораторные эксперименты, которые не должны засорять основной канал продукта.
- Передачи дел, где сохранение контекста важнее, чем мгновенная смекалка модели.

## Что режим Swarm намеренно НЕ делает

- Он не отменяет необходимость одобрения человеком необратимых внешних действий.
- Он не заставляет воркеров общаться с Eric напрямую.
- Он не требует, чтобы каждый воркер всегда работал на одной и той же машине.
- Он не притворяется, что история чата — это система управления проектами.
- Он не исправляет плохие спецификации. Он заставляет плохие спецификации становиться видимыми быстрее. Что менее романтично, но более полезно.
