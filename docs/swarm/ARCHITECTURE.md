# Архитектура Swarm

Режим Swarm построен вокруг надежного цикла: намерение (intent) поступает через Aurora, диспетчеризация (dispatch) проходит через оркестратор, воркеры (workers) выполняются в постоянных сессиях, контрольные точки (checkpoints) возвращаются в плоскость управления, и только решения, заслуживающие оценки (judgment-worthy), доходят до Eric.

## Цикл

```text
┌────────┐
│ Eric   │
└───┬────┘
    │ намерение, оценка, одобрение
    ▼
┌────────────┐
│ Aurora     │
│ главн. агент│
└───┬────────┘
    │ переводит намерение в SwarmBrief
    ▼
┌────────────────────────────┐
│ swarm3 / Оркестратор       │
│ маршрутизация, отклонение,  │
│ эскалация                  │
└───┬────────────────────────┘
    │ диспетчеризация по роли + миссия
    ▼
┌────────────────────────────────────────────────────┐
│ Агенты Hermes                                      │
│ swarm4 исследов. swarm5 сборка  swarm6 проверка     │
│ swarm7 доки      swarm8 опс     swarm9 лаб          │
│ swarm10 патчи    swarm11 QA     swarm12 триаж       │
└───┬────────────────────────────────────────────────┘
    │ контрольная точка с доказательством
    ▼
┌────────────────────────────┐
│ Отчеты / Inbox / рантайм   │
└───┬────────────────────────┘
    │ оркестратор выбирает след. маршрут
    ▼
┌─────────────────────────────────────┐
│ продолжить / исправить / проверить  │
│ / ввод данных                      │
└─────────────────────────────────────┘
```

Ключевое правило: воркеры не пишут Eric сообщения в свободном стиле. Они ставят контрольные точки. Оркестратор маршрутизирует. Aurora берет на себя оценку. Eric одобряет те немногие вещи, которые действительно важны.

## Канонический поток

1. Eric заявляет о результате.
2. Aurora дает название работе и оформляет ее в SwarmBrief.
3. Оркестратор выбирает подходящего воркера или декомпозирует работу.
4. Воркер выполняет задачу внутри своего постоянного профиля и tmux-рантайма.
5. Воркер возвращает каноническую контрольную точку.
6. Роутер уведомлений отправляет контрольную точку оркестратору по умолчанию.
7. Оркестратор решает, стоит ли продолжить, исправить, передать, проверить или эскалировать.
8. Отчеты (Reports) и Входящие (Inbox) делают состояние доступным для проверки.

## Структура SwarmBrief

Канонический YAML находится в разделе 3 файла `SWARM_SPEC.md`. Это его публичная структура:

```yaml
brief_id: brief-<timestamp>-<slug>
worker: swarm<N>
project: <project-name>
goal: <конечный результат одним предложением>
why_now: <триггер>
scope:
  - ограниченный элемент
deliverables:
  - точный путь к артефакту
test_or_proof:
  - команда, проверка, скриншот, артефакт или проверка байтов
constraints:
  - жесткие ограничения
checkpoint_contract:
  state: DONE|HANDOFF|BLOCKED|NEEDS_REVIEW|NEEDS_INPUT
  files_changed: список
  commands_run: список
  proof: тесты/сборка/дым/доказательство проверки
  next_action: точная передача
  blockers: точный блокировщик
escalation:
  on_blocked: маршрут
  on_done: маршрут
budget:
  wall_clock_hours: 2
```

Бриф — это не просто свалка промптов. Это наименьший рабочий контракт, который позволяет воркеру выполнять задачу, не выдумывая рамки проекта.

## Контракт контрольной точки (Checkpoint contract)

Воркеры возвращают этот блок:

```text
STATE: DONE | BLOCKED | NEEDS_INPUT | HANDOFF | IN_PROGRESS | NEEDS_REVIEW
FILES_CHANGED: точные пути или none
COMMANDS_RUN: точные команды или none
RESULT: конкретный результат/доказательство
BLOCKER: блокировщик или none
NEXT_ACTION: точное рекомендуемое следующее действие
```

Хорошие контрольные точки содержат доказательства. Плохие — прилагательные. Swarm оптимизирует для доказательств.

## Маршрутизация уведомлений

Роутер уведомлений находится в `src/server/swarm-notifications.ts`.

Текущее поведение:

- Контрольные точки по умолчанию направляются воркеру-оркестратору.
- Воркер-оркестратор по умолчанию — `swarm3`.
- Цель tmux — `swarm-swarm3`.
- Дубликаты "сырых" контрольных точек подавляются через `runtime.json`.
- `NEEDS_INPUT` эскалируется в основную сессию.
- Если сессия tmux оркестратора недоступна, контрольная точка эскалируется в основную сессию.
- `DONE`, `HANDOFF` и `BLOCKED` сначала идут к оркестратору.
- Основная сессия получает прямую эскалацию только тогда, когда требуется вмешательство человека или когда с оркестратором невозможно связаться.

Это разделение важно. Без него основной чат превращается в свалку мелочей от воркеров. Технический термин.

## Постоянные миссии против разовых заданий

### Постоянные миссии

Постоянная миссия — это бессрочная обязанность воркера. Примеры:

- Scribe ведет документацию и передачи.
- Reviewer владеет этапом проверки с подтверждением байтов.
- Triage работает в канале PR/issues.
- Lab проводит эксперименты с моделями/рантаймом.
- Foundation поддерживает инфраструктуру исправности и ремонта.

Постоянные миссии позволяют простаивающим воркерам оставаться полезными, не дожидаясь, пока Eric придумает им работу.

### Разовые задания (Ad-hoc dispatches)

Разовое задание — это ограниченная задача. Для нее используется тот же профиль, та же роль, тот же формат контрольной точки и тот же Greenlight Gate.

Примеры:

- "Обновить docs/swarm/QUICKSTART.md для нового диалога Add Swarm."
- "Проверить PR #42 и вернуть APPROVED/CHANGES_REQUESTED с доказательством байтов."
- "Воспроизвести issue #17 и написать минимальный провальный тест."

Система должна рассматривать разовые задания как миссии с меньшим радиусом действия, а не как обычные запросы в чате.

## Три постоянных канала (Lanes)

### Канал A — Запуск / демо / творческая сборка

Цель: поставка скоординированных артефактов запуска, демо-версий, медиа и ресурсов для релизов.

Типичные владельцы:

- Builder для реализации
- Mirror Integrations для ресурсов (assets)
- Sage для повествования и исследований
- QA для дымовых тестов (smoke checks)
- Scribe для текстов README/презентаций

### Канал B — Автопилот Issues + PR

Цель: продвижение открытых GitHub issues и PR.

Типичные владельцы:

- Triage как основной исполнитель
- Overflow для подстраховки
- Reviewer для контроля на входе
- QA для проверки на регрессии

Основной цикл:

```text
сканирование -> оценка -> воспроизведение -> патч -> тест -> PR -> проверка -> одобрение человеком
```

### Канал C — Лаборатория / эксперименты

Цель: проведение экспериментов без дестабилизации продуктового канала.

Типичный владелец:

- Lab

Примеры:

- тесты локальных моделей
- сравнение рантаймов
- спекулятивные эксперименты с производительностью
- циклы прототипирования

Lab получает автономность, так как изоляция снижает риски. Продуктовый канал получает доказательства, когда Lab находит что-то реальное.

## Greenlight Gate (Шлюз одобрения)

Swarm может готовить рискованные действия. Он не может совершать их молча.

Требуется явное одобрение человека перед:

- `git push --force`
- слиянием (merge) или закрытием PR
- закрытием issue без явной инструкции
- созданием релиза
- публикацией в npm/pnpm
- публичными постами в X/Discord/блогах
- финансовыми транзакциями
- деструктивными операциями с файлами
- перезапуском основных сервисов

Документация и локальные файлы могут создаваться агрессивно. Внешне видимые действия остаются заблокированными до одобрения.

## Плейбук авторемонта (Auto-repair playbook)

Плейбук ремонта сопоставляет известные режимы сбоев с безопасными исправлениями. Оркестратор должен сверяться с ним перед эскалацией.

Примеры классов ремонта:

- отсутствие сессии tmux
- устаревший рантайм воркера
- несоответствие пути профиля
- сбой сборки/теста с известной командой
- таймаут контрольной точки
- недоступность авторизации/токена
- отклонение ветки (branch drift)

Ремонт ограничен. Если исправление может стать деструктивным, внешне видимым или спекулятивным — эскалируйте.

## Состояние рантайма

У каждого воркера есть запись рантайма с полями:

- ID воркера
- роль
- состояние
- фаза
- текущая задача
- cwd (текущая директория)
- время последнего вывода
- последняя отметка (check-in)
- последняя сводка/результат
- следующее действие
- причина блокировки
- статус контрольной точки
- счетчики задач
- счетчики cron

UI использует это для отрисовки карточек, отчетов (Reports), входящих (Inbox) и целей подключения рантайма.

## Эндпоинты плоскости управления (Control-plane)

Важные эндпоинты:

| Эндпоинт | Цель |
| --- | --- |
| `GET /api/swarm-roster` | Возврат настроенных агентов Hermes и метаданных ролей. |
| `GET /api/swarm-runtime` | Возврат состояния рантайма и возможности подключения к tmux. |
| `GET /api/swarm-missions` | Возврат истории миссий и назначений. |
| `POST /api/swarm-dispatch` | Отправка работы одному или нескольким агентам Hermes. |
| `POST /api/swarm-tmux-start` | Запуск сессии воркера на базе tmux. |
| `POST /api/swarm-tmux-stop` | Остановка сессии воркера tmux. |
| `POST /api/swarm-tmux-scroll` | Прокрутка сессии tmux из UI. |
| `GET /api/swarm-health` | Сводка о состоянии локального Swarm. |

## Философия сбоев

Система должна давать сбои так, чтобы следующий участник точно знал, что делать.

Хороший блокировщик:

```text
BLOCKER: gh auth status failed with missing token; cannot create PR.
NEXT_ACTION: Provide a GitHub token or run gh auth login
```

Плохой блокировщик:

```text
BLOCKER: Something went wrong with the PR.
NEXT_ACTION: Please fix it.
```

## Безопасная операционная граница

Swarm может готовить коммиты, ветки, тексты PR, вердикты проверок, issues и примечания к релизу. Он не должен выполнять слияние, пуш с флагом force, публикацию, публичные анонсы, закрытие issues или деструктивные операции с файлами без явного одобрения человека.

Эта граница и есть Greenlight Gate. Пусть она будет скучной.
