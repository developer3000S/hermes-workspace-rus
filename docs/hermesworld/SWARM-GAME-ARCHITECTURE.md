# Архитектура игры HermesWorld Swarm

Последнее обновление: 2026-05-06

## Основная идея

HermesWorld должна строиться как нативная для агентов игровая студия:

- один оркестратор владеет направлением продукта и интеграцией
- воркеры (исполнители) владеют ограниченными системами
- каждое направление поставляет артефакты, подтверждающие результат
- интеграция происходит через небольшие PR/патчи, а не через одну гигантскую ветку
- безопасность призов/пасхалок обеспечивается на стороне сервера и является приватной

## Направление рантайма

- Hermes Workspace встраивает хостинг HermesWorld через iframe.
- Рантайм HermesWorld живет на `hermes-world.ai`.
- Workspace остается OSS-оболочкой/дистрибутивом.
- Игровая логика и логика, чувствительная к призам, переносится в хостинговые/приватные сервисы.

## Текущий практический стек

- React + TypeScript
- Three.js / React Three Fiber
- Экосистема Drei
- Cloudflare Pages + Workers
- WebSocket-реле для мультиплеера
- localStorage/IndexedDB for для простых локальных сохранений сюжета
- приватный призовой оракул для ценных клеймов (запросов на получение)

## Направления (лейны) Swarm

### Направление A — Shell/мобильный UX

Цель: сделать HermesWorld играбельной на телефонах.

Результаты:
- фиксированный вьюпорт, отсутствие прокрутки документа
- макет мобильного HUD "островного" типа
- сворачиваемые цели/чат
- актуальные заметки по дымовому тестированию на мобильных устройствах

Условие остановки:
- сборка проходит, а скриншот/описание доказывают отсутствие регрессий с прокруткой или загромождением экрана

### Направление B — Системы мира/геймплея

Цель: сделать игровой цикл реалистичным.

Результаты:
- границы систем квестов/событий
- модель диалогов/состояний NPC
- очистка системы инвентаря/титулов/наград
- структура API действий агента

Условие остановки:
- минимальный патч или спецификация с точными файлами и интерфейсами

### Направление C — Контент/пасхалки

Цель: обеспечить исследуемость и лор без утечки призов.

Результаты:
- публичный слой лора/пасхалок
- интерфейс приватного призового оракула
- дизайн охоты за 7 сигилами (печатями)
- граница между приманками и данными, чувствительными к призам

Условие остановки:
- отсутствие жестко закодированных координат призов/секретов в клиенте

### Направление D — Инфраструктура/мультиплеер/призовой оракул

Цель: серверный авторитарный бэкенд.

Результаты:
- архитектура Cloudflare Worker
- контракт эндпоинта клеймов (claim)
- поток подписанных сообщений кошелька
- модель античита/рейт-лимитов/логов событий

Условие остановки:
- только спецификация приватного сервиса и заглушки, если секреты недоступны

### Направление E — Арт/ассеты/процедурная генерация

Цель: быстрее улучшить внешний вид зон.

Результаты:
- рекомендации по пайплайну ассетов
- гайд по стилю glTF/low-poly
- библиотека промптов для генерации ассетов через Meshy/Tripo/Spline/Codex
- план визуального прохода по зонам

Условие остановки:
- 5 конкретных инструментов/промптов/ассетов для тестирования на этой неделе

### Направление F — Ревьюер интеграции

Цель: предотвратить хаос роя (swarm).

Результаты:
- проверка каждого направления на предмет пересечений/безопасности/производительности
- принудительное использование небольших патчей
- отклонение секретов на стороне клиента
- рекомендации по порядку слияния (merge)

## Правила интеграции

1. Никаких секретов призов в клиентском коде.
2. Никаких гигантских PR, если это не одобрено явно.
3. Каждый воркер сообщает:
   - измененные файлы
   - запущенные команды
   - результат
   - блокирующие факторы
   - следующее действие
4. Основная сессия принимает решения о слиянии.
5. Мобильный UX должен тестироваться как ограничение вьюпорта, а не угадываться.
