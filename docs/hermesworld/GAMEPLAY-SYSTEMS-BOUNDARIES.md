# Границы игровых систем HermesWorld (Направление B)

Статус: контракт реализации для параллельных воркеров роя (swarm workers)
Область: квесты, события, состояние NPC, инвентарь, награды и API действий агентов
Источник: `docs/hermesworld/SWARM-GAME-ARCHITECTURE.md`

## Цель

Сделать игровой цикл реалистичным, не создавая гигантскую общую ветку. Воркеры должны вести разработку за стабильными "швами" (seams) и изменять только те файлы, которыми владеет их направление.

## Карта владения

| Система | Файлы владельца | Разрешен импорт из | Не должен владеть |
| --- | --- | --- | --- |
| Контракты | `src/screens/playground/lib/gameplay-contracts.ts` | нет / только типы | Мутация состояния рантайма |
| Движок квестов | `src/screens/playground/lib/quest-engine.ts`, тесты квестов | `gameplay-contracts`, статические данные квестов | Хуки React, тексты NPC, UI инвентаря |
| Шина/лог событий | `src/screens/playground/lib/gameplay-events.ts`, тесты событий | `gameplay-contracts` | Определения квестов, расчет наград |
| Состояние/диалоги NPC | `src/screens/playground/lib/npc-state.ts`, `npc-dialog.ts` | `gameplay-contracts`, эмиттеры событий | Мутация инвентаря, выдача наград |
| Инвентарь/награды | `src/screens/playground/lib/inventory-rewards.ts`, тесты наград | `gameplay-contracts`, данные предметов | Решения о завершении квестов, тексты диалогов NPC |
| Действия агентов | `src/screens/playground/lib/agent-actions.ts`, заглушки API позже | `gameplay-contracts`, шина событий | Призовой оракул, прямые секреты, оркестрация UI |
| Интеграция с React | `src/screens/playground/hooks/use-playground-rpg.ts` | все модули сервисов | Новые бизнес-правила после их извлечения |

Правило: статический контент может жить в `playground-rpg.ts`/`npc-dialog.ts`; игровые правила должны быть перенесены в небольшие чистые модули с тестами.

## Граница рантайма

Использовать один путь перехода состояния в стиле reducer:

1. UI/NPC/агент порождает `GameplayEvent` или `AgentActionRequest`.
2. Событие/действие валидируется относительно текущего `GameplayStateSnapshot`.
3. Чистая система возвращает `GameplayPatch` плюс опциональные последующие события.
4. Хук React применяет патчи и отображает уведомления (toasts).
5. Хостинговые/серверные маршруты могут позже воспроизвести те же события для авторитарных потоков мультиплеера/призов.

Не позволяйте компонентам напрямую выдавать предметы, завершать квесты или разблокировать миры, кроме как через фасад, предоставляемый `usePlaygroundRpg`.

## Контракт событий

События — это факты, которые уже произошли. Они должны быть только для добавления (append-only) и сериализуемыми.

Необходимые события MVP:
- `npc.talked` — игрок начал диалог с NPC.
- `npc.choice_selected` — игрок выбрал вариант диалога.
- `quest.objective_completed` — выполнение цели принято движком квестов.
- `quest.completed` — все обязательные цели приняты.
- `inventory.item_granted` — предмет добавлен (идемпотентно).
- `inventory.item_equipped` — изменен слот снаряжения.
- `world.entered` — игрок вошел/разблокировал зону.
- `chat.sent` — отправлено сообщение в локальный/общий чат.
- `combat.enemy_defeated` — победа над врагом принята.
- `agent.action_requested` / `agent.action_completed` / `agent.action_failed` — жизненный цикл действия агента.

Каждое событие должно включать `id`, `type`, `createdAt`, `actorId` и `source`. Обработчики событий должны быть идемпотентными, так как воспроизведение в мультиплеере/на сервере будет дублировать пакеты. Забавно, но распределенные системы в конечном итоге превращают любую игру в бухгалтерский учет.

## Граница квестов

Движок квестов владеет:
- сопоставлением целей (objectives) из событий
- завершением обязательных и опциональных целей
- идемпотентностью завершения квеста
- порождением намерений на награду (reward intents), а не прямой мутацией инвентаря

Движок квестов НЕ владеет:
- текстами уведомлений (toasts)
- текстами диалогов NPC
- определениями предметов
- визуальным порядком в HUD
- секретами призов/пасхалок

Форма MVP API:

```ts
advanceQuests(snapshot, event): QuestAdvanceResult
```

`QuestAdvanceResult` возвращает id выполненных целей, id завершенных квестов и намерения `RewardGrant[]`. Система инвентаря/наград применяет эти намерения.

## Граница состояния NPC

Состояние NPC владеет:
- флагами отношений для каждого NPC
- id просмотренных узлов диалога
- предикатами доступности выбора
- обеспечением кулдауна/одноразового выбора
- генерацией событий для выбранных вариантов

Состояние NPC НЕ владеет:
- добавлением предметов в инвентарь
- прямым завершением квестов
- прямой разблокировкой миров

Варианты выбора в диалогах NPC должны перейти от императивных полей типа `grantItems`/`completeQuest` к декларативным `effects: GameplayEffect[]`. Во время миграции сохраняйте старые поля, но нормализуйте их до эффектов на границе системы.

## Граница инвентаря и наград

Инвентарь/награды владеет:
- идемпотентной выдачей предметов
- семантикой стаков/валюты при добавлении
- валидацией слотов снаряжения
- применением XP навыков / титулов / разблокировкой миров
- дескрипторами уведомлений о наградах

Инвентарь/награды НЕ владеет:
- принятием решений о завершении квеста
- принятием решений о доступности выбора у NPC
- вызовом внешних призовых сервисов

Правило MVP: держите id предметов публичными и безвредными. Все ценное использует поток получения (claim) через хостинг-оракул призов из Направления D, никогда через клиентский инвентарь.

## Граница API действий агентов

Действия агентов — это игровые глаголы (действия), которые позже могут вызывать Hermes. Они должны быть объектами запрос/ответ, а не случайными (ad hoc) колбэками компонентов.

Действия MVP:
- `agent.ask_npc` — получение динамической реплики/лора от NPC.
- `agent.build_prompt` — превращение текста игрока в резюме артефакта Кузницы (Forge).
- `agent.summon_companion` — спавн временных метаданных фамильяра.
- `agent.judge_duel` — оценка результата дуэли промптов на Арене.
- `agent.generate_lore` — создание не связанного с призами лора/атмосферы зоны.

Обработчик действий агента владеет:
- валидацией схемы
- метаданными лимитов частоты запросов (rate-limit)
- сборкой безопасного публичного контекста
- событиями статуса

Обработчик действий агента НЕ должен владеть:
- секретами / решениями о призах
- прямой мутацией состояния без возврата `GameplayPatch`
- выбором "сырых" моделей/провайдеров внутри компонентов UI

## Порядок реализации MVP

1. `gameplay-contracts.ts` — только общие типы. Минимальная вероятность конфликтов.
2. `gameplay-events.ts` — фабрика событий + помощники для идемпотентности.
3. `inventory-rewards.ts` — чистые функции применения наград; вынос дублирующегося кода наград из хука.
4. `quest-engine.ts` — чистая логика продвижения по целям/квестам на основе событий.
5. `npc-state.ts` — нормализация эффектов диалогов, одноразовый выбор, флаги отношений.
6. `agent-actions.ts` — сначала локальный мок-обработчик; позже вызовы сервера/Hermes.
7. Тонкая интеграция `usePlaygroundRpg` — хук становится оркестрацией/склейкой, а не "кашей" из игровых правил.

## Правила для параллельных воркеров

- Один воркер на каждый модуль выше.
- Каждый воркер добавляет теги для своего чистого модуля перед тем, как трогать React.
- Ни один воркер не правит код призов/клеймов, если ему не назначено Направление D.
- Ни один воркер не меняет визуальный макет при извлечении систем, если ему не назначено Направление A.
- Изменения общих типов происходят только в `gameplay-contracts.ts` и должны быть обратно совместимы по умолчанию.
- Отдавайте предпочтение добавочным адаптерам, а не переписыванию `playground-rpg.ts` за один проход.

## Критерии приемки

- Новые системы сериализуемы и детерминированы.
- Повторное применение одного и того же события не выдает награду дважды.
- Квесты порождают намерения на награду; инвентарь их применяет.
- Выбор NPC порождает эффекты/события; он не мутирует состояние профиля напрямую.
- Действия агентов возвращают патчи/события и никогда не встраивают секреты в состояние клиента.
- `pnpm build` проходит успешно после каждого патча воркера.
