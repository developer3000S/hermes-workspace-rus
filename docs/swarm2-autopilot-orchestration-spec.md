# Спецификация оркестрации автопилота Swarm2

Дата: 28.04.2026
Статус: Спецификация реализации, поэтапное развертывание
Канонический репозиторий: `/Users/system/Documents/mybuild/hermes-workspace-rus`

## Цель

Swarm2 должен вести себя как постоянная команда на автопилоте, а не просто как красивый интерфейс для нескольких чатов.

Пользователь должен иметь возможность сказать оркестратору:

> Настрой четырех агентов: один занимается PR, один проводит исследования, один создает функции, один проверяет (ревью). Запусти эту миссию.

Затем Swarm2 должен:

1. создать или обновить список роя (roster),
2. назначить роли, навыки и миссии,
3. направить работу в соответствующие постоянные сессии воркеров Hermes,
4. поддерживать работу каждого воркера в его собственном профиле / tmux-сессии,
5. собирать контрольные точки (checkpoints) по мере выполнения воркерами подзадач,
6. автоматически давать воркерам промпты для продолжения работы, если задачи еще остались,
7. проверять результаты перед тем, как считать рабочий процесс завершенным,
8. отображать весь цикл в `/swarm2`, не требуя постоянного ручного контроля.

## Принципы продукта

- **Постоянные агенты, а не одноразовые помощники.** Каждый воркер обладает идентичностью, памятью профиля, состоянием среды выполнения и живой сессией.
- **Список (Roster) — источник истины.** Эвристика по номерам воркеров — только запасной вариант. Маршрутизация читает `swarm.yaml`.
- **Доказательства важнее ощущений.** Воркеры фиксируют измененные файлы, запущенные команды, результаты, блокировщики и следующее действие.
- **Оркестратор управляет циклом.** Воркеры исполняют. Оркестратор декомпозирует, выстраивает последовательность, мониторит, выдает новые промпты, проверяет и эскалирует проблемы.
- **Поэтапный автопилот.** Система должна работать в ручном, полуавтоматическом, а затем в полностью автоматическом режимах.
- **Нативно для tmux/Claude.** Сессии воркеров — это профили Hermes в tmux-сессиях `swarm-<workerId>`.

## Текущий фундамент

Уже присутствует:

- `swarm.yaml`, канонический конфиг списка.
- `/api/swarm-roster`, чтение/запись записей списка.
- `/api/swarm-decompose`, разбиение миссии на задания.
- `/api/swarm-dispatch`, отправка промптов в живые сессии tmux/Claude с откатом к разовым запросам Claude.
- `/api/swarm-runtime`, чтение `runtime.json` воркера, состояния tmux, задач, артефактов, предпросмотров.
- `/api/swarm-chat`, чтение истории чата воркера из `state.db` профиля.
- `/api/swarm-tmux-start/stop/scroll`, управление постоянными сессиями.
- Навыки Swarm:
  - `swarm-worker-core`
  - `swarm-orchestrator`
  - `swarm-dev-runtime`
  - `swarm-ui-worker`
  - `swarm-pr-worker`
  - `swarm-bench-worker`

## Целевая архитектура

```text
Миссия пользователя
  ↓
Оркестратор
  ↓ читает/записывает
swarm.yaml (список) + runtime (состояние)
  ↓
Декомпозитор/роутер
  ↓ производит
План заданий (Assignment plan)
  ↓
Диспетчер
  ↓ промпт для каждого воркера в tmux/Claude
Постоянные сессии воркеров
  ↓ воркеры ставят контрольные точки
runtime.json + state.db + артефакты/предпросмотры
  ↓
Цикл оркестратора
  ↓
продолжить / перенаправить / проверить / завершить / эскалировать
```

## Контракт списка (Roster)

`swarm.yaml` остается человекочитаемым и переносимым. Каждая запись воркера должна поддерживать:

```yaml
workers:
  - id: swarm5
    name: Swarm5
    role: Builder (Строитель)
    specialty: full-stack реализация и интеграция UI/систем
    model: GPT-5.5
    mission: Выпуск сфокусированных частей продукта в Hermes Workspace с тестами.
    skills: [swarm-ui-worker, swarm-worker-core]
    capabilities:
      - code-editing
      - ui-implementation
      - build-verification
    defaultCwd: /Users/system/Documents/mybuild/hermes-workspace-rus
    preferredTaskTypes: [implementation, refactor, ui]
    maxConcurrentTasks: 1
    acceptsBroadcast: true
    reviewRequired: true
```

### Необходимые поля (сейчас)

- `id`
- `name`
- `role`
- `specialty`
- `model`
- `mission`
- `skills`

### Поля для добавления в будущем

- `capabilities`
- `defaultCwd`
- `preferredTaskTypes`
- `maxConcurrentTasks`
- `acceptsBroadcast`
- `reviewRequired`

## Контракт среды выполнения (Runtime)

Каждый профиль воркера может иметь файл `~/.hermes/profiles/<workerId>/runtime.json`.

Оркестратор и воркер должны поддерживать эти поля в актуальном состоянии:

- `workerId`
- `role`
- `state`: `idle | executing | thinking | writing | waiting | blocked | syncing | reviewing | offline`
- `phase`
- `currentTask`
- `activeTool`
- `cwd`
- `lastCheckIn`
- `lastSummary`
- `lastResult`
- `nextAction`
- `blockedReason`
- `checkpointStatus`: `none | in_progress | done | blocked | handoff | needs_input`
- `needsHuman`
- `lastDispatchAt`
- `lastDispatchMode`
- `lastDispatchResult`
- `tasks[]`
- `artifacts[]`
- `previews[]`

## Контракт заданий (Assignment)

`/api/swarm-decompose` должен возвращать задания в таком формате:

```json
{
  "assignments": [
    {
      "workerId": "swarm4",
      "task": "Изучить варианты и подготовить краткую рекомендацию с источниками.",
      "rationale": "Линия исследований отвечает за технический синтез.",
      "expectedOutput": "Рекомендация с описанием компромиссов и следующим действием.",
      "dependsOn": [],
      "reviewRequired": false
    }
  ],
  "unassigned": []
}
```

На этапе 1 требуются только `workerId`, `task`, `rationale`.

## Контракт диспетчера (Dispatch)

`/api/swarm-dispatch` должен принимать либо старый формат широковещательной рассылки:

```json
{ "workerIds": ["swarm1"], "prompt": "..." }
```

либо формат заданий:

```json
{
  "assignments": [
    { "workerId": "swarm4", "task": "...", "rationale": "..." },
    { "workerId": "swarm5", "task": "...", "rationale": "..." }
  ]
}
```

Диспетчеризация заданий должна отправлять каждому воркеру только его задачу плюс компактный контекст оркестрации.

## Конверт промпта воркера

Каждая отправленная задача должна быть обернута в:

1. контекст оркестратора,
2. идентификацию воркера из `swarm.yaml`,
3. названия стека навыков,
4. текущую задачу,
5. контракт контрольной точки / отчета,
6. инструкцию по продолжению.

Пример:

```text
## Swarm Orchestrator Dispatch
Воркер: swarm5 — Builder
Миссия: Выпуск сфокусированных частей продукта в Hermes Workspace с тестами.
Навыки: swarm-ui-worker, swarm-worker-core

## Назначенная задача
...

## Требуемый формат контрольной точки
Ответьте / отметьтесь в формате:
STATE: DONE | BLOCKED | NEEDS_INPUT | HANDOFF | IN_PROGRESS
FILES_CHANGED: ...
COMMANDS_RUN: ...
RESULT: ...
BLOCKER: ...
NEXT_ACTION: ...

Если это одна задача в большом рабочем процессе, остановитесь после контрольной точки и ждите продолжения от оркестратора.
```

## Цикл оркестратора

Цикл оркестратора работает на основе состояния рантайма:

1. сбор состояний воркеров из `/api/swarm-runtime`,
2. проверка недавнего чата из `/api/swarm-chat`,
3. пометка воркеров как устаревших (stale), если `lastCheckIn` слишком старый,
4. обнаружение блокировщиков,
5. обнаружение контрольных точек завершения,
6. назначение следующей задачи, если работа осталась,
7. направление задач на проверку в соответствующие линии ревьюеров,
8. эскалация проблемы пользователю только при необходимости его участия.

### Состояния цикла

- `planning`: строится декомпозиция миссии.
- `dispatching`: рассылаются начальные задания.
- `executing`: воркеры активны.
- `checkpointing`: воркеры возвращают доказательства выполнения.
- `reviewing`: ревьюер/оркестратор проверяет результаты.
- `continuing`: отправляются следующие задачи.
- `blocked`: требуется вмешательство человека или изменение среды.
- `complete`: финальный результат готов к передаче.

## Этапы автопилота

### Этап 1 (реализуется сейчас)

- Роутер использует реальные метаданные списка вместо жестко закодированных ролей по номерам.
- Декомпозитор получает полный контекст списка.
- Диспетчер принимает задания для конкретных воркеров.
- Диспетчер записывает начальную контрольную точку рантайма.
- Промпты воркеров включают контракт навыков/контрольных точек.

### Этап 2

- Добавление `/api/swarm-checkpoint` для безопасной записи структурированных контрольных точек воркеров.
- Парсинг сообщений контрольных точек воркеров из чата и обновление состояния рантайма.
- Добавление ID заданий и ID родительских миссий.

### Этап 3

- Добавление `/api/swarm-orchestrator-loop`.
- Цикл отслеживает рантайм/чат и автоматически продолжает работу воркеров.
- Добавление обнаружения зависаний и отклонений.
- Добавление маршрутизации в линии ревьюеров.

### Этап 4

- Добавление истории миссий для пользователя.
- Добавление сохраненных рабочих процессов / шаблонов.
- Добавление многопользовательской начальной настройки (bootstrapping), чтобы при новой установке агенты и роли создавались бесшовно.

## Дымовой тест Этапа 1

Миссия:

> Протестировать автопилот Swarm2. Изучить одно улучшение, реализовать один небольшой безопасный артефакт, проверить результат.

Ожидаемый маршрут:

- Воркер-исследователь (Research) получает задачу на исследование/контрольную точку.
- Строитель (Builder) получает задачу на реализацию/контрольную точку.
- Ревьюер (Reviewer) получает задачу на проверку/контрольную точку.

Критерии прохождения:

- `/api/swarm-decompose` возвращает JSON заданий, используя роли из списка.
- `/api/swarm-dispatch` возвращает `delivery: tmux` для живых воркеров.
- `runtime.json` для каждой цели показывает `state: executing`, текущую задачу и `checkpointStatus: in_progress`.
- В чате карточки воркера со временем появляется отправленное сообщение.
- Как минимум один воркер отвечает сообщением, содержащим доказательства состояния.

## Открытые вопросы

- Должны ли новые агенты, добавленные только в список, автоматически создавать профили и обертки Hermes, или кнопка "Добавить Swarm" должна оставаться только в конфиге до первого запуска?
- Где должен запускаться цикл оркестратора: в сессии браузера, интервалом на сервере, заданием cron или в постоянном воркере-оркестраторе Claude?
- Насколько агрессивным должно быть автоматическое продолжение работы до того, как спросить пользователя?
- Должна ли проверка быть обязательной только для задач, изменяющих код, или для всех рабочих процессов?
