# Спецификация жизненного цикла воркеров Swarm2 и компактации (Compaction)

Дата: 28.04.2026
Статус: поэтапная реализация

## Проблема

Воркеры Swarm — это постоянные агенты Claude со своими профилями, сессиями, панелями tmux, состоянием среды выполнения и памятью. Это правильная архитектура, но при длительной работе воркеров качество их ответов неизбежно падает, когда объем контекста приближается к лимиту окна модели.

Нам нужна автоматическая система управления жизненным циклом, чтобы воркеры могли:

1. работать с большими лимитами контекста,
2. создавать контрольные точки (checkpoint) до того, как качество упадет,
3. записывать надежные отчеты о передаче дел (handoffs),
4. чисто перезапускаться / обновляться,
5. возобновлять работу на основе отчета и состояния миссии,
6. держать оркестратора в курсе.

## Целевое поведение

Для каждого воркера устанавливается политика контекста:

- мягкий лимит: 250к токенов — вскоре запросить краткую контрольную точку;
- лимит передачи: 400к токенов — записать полный отчет о передаче до продолжения работы;
- жесткий лимит: 500к токенов — прекратить прием новых задач до обновления.

Точные цифры должны настраиваться для каждой модели/профиля, но стандартная политика должна быть безопасной.

## Состояния жизненного цикла

- `healthy` (здоров): ниже мягкого лимита;
- `watch` (наблюдение): выше мягкого лимита, продолжаем работу, но мониторим;
- `handoff_required` (требуется передача): выше лимита передачи, просим воркера записать отчет;
- `renew_required` (требуется обновление): выше жесткого лимита или воркер постоянно "лагает"/фрагментирован;
- `renewing` (обновление): запрошен отчет, сессия tmux/Claude перезапускается;
- `blocked` (заблокирован): ошибка обновления или отсутствие отчета.

## Контракт отчета о передаче (Handoff)

Перед обновлением воркер должен записать или вернуть отчет, содержащий:

```text
STATE: HANDOFF
FILES_CHANGED: ... (измененные файлы)
COMMANDS_RUN: ... (запущенные команды)
RESULT: ... (текущее состояние и что было сделано)
BLOCKER: ... (блокировщик или "нет")
NEXT_ACTION: ... (точный следующий шаг после обновления)
```

Путь к постоянному отчету:

```text
/Users/system/.openclaw/workspace/memory/handoffs/swarm/<workerId>-latest.md
```

Опционально могут существовать архивные копии с метками времени, но `latest.md` является источником для возобновления.

## Последовательность обновления (Renewal)

1. Обнаружение давления контекста по количеству токенов сессии в `state.db` Claude.
2. Запрос отчета у воркера через диспетчеризацию в tmux.
3. Парсинг контрольной точки отчета из чата.
4. Сохранение отчета в постоянную память.
5. Остановка сессии tmux воркера.
6. Запуск чистой сессии Claude с тем же профилем и в той же рабочей директории.
7. Отправка промпта возобновления, содержащего сводку отчета + активное задание миссии.
8. Пометка состояния рантайма как `healthy/executing`.

## Требования к продукту

Интерфейс Swarm2 должен показывать:

- состояние контекста для каждого воркера;
- текущую оценку токенов сессии;
- статус жизненного цикла;
- время последнего отчета о передаче;
- кнопку обновления;
- статус автоматического обновления.

## Правила безопасности

- Никогда не обновлять автоматически, пока воркер активно пишет, если только не достигнут жесткий лимит.
- Никогда не начинать деструктивное выполнение после обновления без разрешения политики миссии.
- Отчет о передаче должен быть завершен до перезапуска, если только человек не инициирует принудительное обновление.
- Если парсинг отчета не удался, пометить воркера как `blocked` и запросить помощь оркестратора/человека.

## Реализация этапа 1

- Добавление API статуса жизненного цикла.
- Чтение последних показателей количества токенов из `state.db`.
- Возврат состояния жизненного цикла и рекомендуемого действия.
- Добавление действия `request-handoff`, которое отправляет строгий промпт для отчета в tmux.
- Добавление действия `renew` (обновление), но пока с требованием `force: true`.
- Нормализация оберток swarm для использования текущей директории проекта в качестве рабочей.

## Этап 2

- Парсинг контрольных точек отчета в постоянные файлы отчетов.
- Добавление полей `contextTokens`, `contextState`, `lastHandoffAt` в `runtime.json`.
- Добавление бейджей жизненного цикла в UI Swarm2.

## Этап 3

- Цикл автоматического обновления.
- Промпт возобновления с состоянием миссии.
- Политики контекста для конкретных моделей.
