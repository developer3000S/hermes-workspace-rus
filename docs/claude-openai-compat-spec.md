# Спецификация архитектуры совместимости Hermes Workspace с OpenAI

> **Для Claude:** Используйте `writing-plans`, если это превратится в план реализации. Этот документ фиксирует направление совместимости продукта и бэкенда.

**Цель:** Сделать так, чтобы Hermes Workspace работал "из коробки" с ванильным `hermes-agent` и любым OpenAI-совместимым бэкендом, автоматически разблокируя расширенные функции рабочего пространства при наличии специфических API Claude.

**Статус:** Утвержденное архитектурное ограничение для следующего этапа реализации.

---

## 1. Проблема

В настоящее время Hermes Workspace зависит от форка шлюза `hermes-agent` для обеспечения расширенной функциональности:

- управление сессиями
- потоковый чат
- браузер памяти
- браузер навыков / процесс установки
- редактирование конфигурации
- поведение дашборда с учетом возможностей

Зависимость от этого форка — неверная форма для дистрибуции.

Текущие недостатки:

- пользователи не могут направить рабочее пространство на стандартный `hermes-agent` и ожидать, что оно заработает
- README и процесс настройки навязывают использование кастомного форка
- надежность чата привязана к `/api/sessions` вместо более переносимого OpenAI-совместимого интерфейса чата
- принятие продукта ограничено политикой бэкенда, а не удобством фронтенда

Мы хотим это изменить.

---

## 2. Архитектурное ограничение

Это решение, которое необходимо закрепить:

> **Hermes Workspace должен работать автономно с любым OpenAI-совместимым бэкендом.**
>
> Специфические функции Claude могут улучшать опыт при наличии полного API Hermes Agent, но продукт должен оставаться пригодным для использования и без этих эндпоинтов.

Не подлежащее обсуждению следствие:

- **Форк не может оставаться обязательным требованием для продукта.**
- Расширенные API — это опциональные возможности, а не предварительные условия для запуска.

---

## 3. Двухэтапная стратегия

### Шаг 1 — Сделать Workspace автономным прямо сейчас

Переписать рабочее пространство так, чтобы основной чат работал с:

- ванильным `hermes-agent`
- любым бэкендом, предоставляющим `/v1/chat/completions`
- любым бэкендом, опционально предоставляющим `/v1/models`

В этом режиме расширенные функции плавно деградируют при отсутствии специфических API Claude.

### Шаг 2 — Перенос расширенного API в основную ветку (upstream) позже

Отправить кастомные эндпоинты Claude в основной репозиторий `hermes-agent`, ориентируясь на `gateway/platforms/api_server.py`.

Если upstream примет их:

- полная функциональность рабочего пространства будет работать с ванильным `hermes-agent`
- не останется долгосрочной зависимости от форка
- улучшенный UX станет первоклассной возможностью основной ветки, а не приватным набором патчей

---

## 4. Режимы продукта

Рабочее пространство должно работать в двух режимах выполнения.

### Режим A — Переносимый OpenAI-совместимый режим

Минимально необходимый набор интерфейсов бэкенда:

- `POST /v1/chat/completions`
- опционально `GET /v1/models`

Пользователь получает:

- работающий чат
- потоковые ответы ассистента, когда бэкенд поддерживает стриминг
- выбор модели, если существует `/v1/models`
- базовые вложения, если бэкенд/модель их поддерживают
- чистый процесс онбординга и статус подключения

Пользователю **НЕ** требуются:

- `/api/sessions`
- `/api/skills`
- `/api/memory`
- `/api/config`
- специфические метаданные Claude

### Режим B — Расширенный режим Claude

При наличии специфических эндпоинтов Claude разблокируются:

- история сессий и именованные сессии
- браузер памяти / поиск / редактирование
- браузер навыков / установка / управление
- редактор конфигурации
- видимость работ (jobs) и cron
- расширенная интроспекция возможностей и рабочего пространства

Интерфейс должен определять эти возможности и прогрессивно улучшаться.

---

## 5. Основной принцип продукта

**Чат — это базовый продукт. Все остальное — опциональные улучшения.**

Если пользователь направляет Hermes Workspace на валидный OpenAI-совместимый бэкенд, он должен иметь возможность отправить сообщение и получить потоковый ответ, не заботясь о том, является ли бэкенд Claude, OpenAI, OpenRouter, Ollama, vLLM или чем-то еще.

Все, что выходит за рамки этого, должно рассматриваться как дополнение на основе возможностей.

---

## 6. Требуемые изменения в поведении

### 6.1 Транспорт чата

Рабочее пространство должно перестать рассматривать `/api/sessions` как обязательное условие для отправки сообщения в чат.

Вместо этого:

1. Определить, существуют ли API сессий Claude.
2. Если да, использовать расширенный поток сессий Claude.
3. Если нет, отправить чат через `POST /v1/chat/completions`.
4. Если стриминг поддерживается, отрисовывать потоковые дельты.
5. Если стриминг не поддерживается, чисто отрисовывать стандартный непотоковый ответ.

Результат:

- отсутствие API сессий Claude больше не должно приводить к зависанию продукта или критической ошибке при базовом чате.

### 6.2 Определение возможностей

Проверка возможностей должна явно различать:

#### Базовые переносимые возможности

- здоровье (health) / доступность
- `/v1/chat/completions`
- `/v1/models`
- поддержка стриминга, если она определима
- поддержка вложений / изображений, если она выводима

#### Улучшения Claude

- `/api/sessions`
- `/api/skills`
- `/api/memory`
- `/api/config`
- `/api/jobs`

Приложение должно предоставлять их в виде двух слоев:

- `coreCapabilities` (базовые возможности)
- `enhancedCapabilities` (расширенные возможности)

### 6.3 Плавная деградация

При отсутствии специфических API Claude интерфейс не должен показывать сломанные индикаторы загрузки, неработающие вкладки или непонятные ошибки.

Вместо этого каждый расширенный раздел должен выполнять одно из следующих действий:

- скрываться, когда он не актуален
- показывать четкое состояние "Недоступно на данном бэкенде"
- объяснять, какая возможность разблокирует его
- продолжать сохранять остальную часть приложения полностью работоспособной

Требуемые состояния деградации:

- **Сессии:** возврат к эфемерному/локальному состоянию ветки чата
- **Память:** состояние "недоступно только для чтения" с объяснением
- **Навыки:** состояние "недоступно" с объяснением
- **Конфигурация:** состояние "недоступно" с объяснением
- **Работы (Jobs):** состояние "недоступно" с объяснением

### 6.4 Онбординг и настройка

Процесс настройки должен перестать инструктировать пользователей о необходимости форка.

Новый принцип настройки:

- сначала подключите любой OpenAI-совместимый бэкенд
- убедитесь, что чат работает
- затем рекламируйте дополнительные нативные функции Claude, если они поддерживаются

Текст онбординга должен сообщать:

- "Работает с любым OpenAI-совместимым бэкендом"
- "Расширенные функции разблокируются автоматически с помощью API шлюза Hermes Agent"

### 6.5 Документация

README и документация по настройке должны честно отражать архитектуру.

Необходимые сообщения:

- рабочее пространство работает автономно с OpenAI-совместимыми бэкендами
- ванильный `hermes-agent` является поддерживаемой целью
- расширенный API Hermes Agent опционален для продвинутых функций рабочего пространства
- перенос этих API в основную ветку — это долгосрочный путь

## 7. Требования к UX

### 7.1 Язык статуса подключения

Не представляйте отсутствие расширенных API как фатальную ошибку, когда базовый чат работает.

Используйте формулировки статуса вроде:

- **Подключено** — чат доступен
- **Расширено** — обнаружены API рабочего пространства Claude
- **Частично** — чат доступен, некоторые расширенные функции недоступны
- **Отключено** — подходящий бэкенд для чата не обнаружен

### 7.2 Ограничение функций (Feature gating)

Ограничение функций должно казаться намеренным, а не поломкой.

Хорошие примеры:

- "Браузер памяти требует API памяти Claude."
- "История сессий еще не доступна на этом бэкенде."
- "Подключено в переносимом режиме. Чат работает; расширенные инструменты рабочего пространства недоступны."

Плохие примеры:

- сырой текст 404
- вечный спиннер
- общие баннеры 500 без указания следующего шага
- экран запуска, утверждающий, что настройка не завершена, когда чат на самом деле пригоден для использования

### 7.3 Поведение сессий в переносимом режиме

Когда API сессий Claude не существует, приложению все равно нужен вменяемый UX чата.

Минимум переносимого режима:

- поддерживать текущую ветку в состоянии клиента
- сохранять видимую историю сообщений для активной страницы/сессии приложения
- четко помечать её как локальную / временную, если сохранение недоступно
- избегать поддельных серверных ID сессий, когда бэкенд их не предоставляет

## 8. Направление проектирования API

### 8.1 Переносимый путь

Основная цель переносимого запроса:

- `POST /v1/chat/completions`

Ожидаемая совместимость запроса:

- `model`
- `messages`
- `stream`
- `temperature`, если поддерживается
- вложения / содержимое изображений, если бэкенд принимает мультимодальные сообщения в стиле OpenAI

Ожидаемая обработка ответа:

- чанки потока SSE для потокового режима
- стандартный JSON ответа чата OpenAI для непотокового режима

### 8.2 Расширенный путь Claude

Расширенный путь остается нативным для Claude там, где он доступен, потому что он обеспечивает:

- постоянные сессии
- историю сообщений
- интерфейсы памяти/навыков/конфигурации
- более богатые возможности рабочего пространства

Это нормально, но это должно находиться за определением возможностей, а не подразумеваться по умолчанию.

### 8.3 Цель для upstream

Для Шага 2 кастомные эндпоинты API должны быть предложены в основную ветку в:

- `gateway/platforms/api_server.py`

Намерение:

- сделать расширенные API рабочего пространства частью основного `hermes-agent`
- избавить от постоянного бремени поддержки вечного форка
- позволить Hermes Workspace относиться к стандартному Claude как к лучшему бэкенду, не делая его обязательным

---

## 9. Что НЕ является целью

Эта спецификация **НЕ** требует:

- универсального паритета между всеми OpenAI-совместимыми провайдерами
- гарантированного сохранения сессий на бэкендах, отличных от Hermes
- поддержки памяти/навыков/конфигурации вне Claude
- создания абстракции бэкенда для каждого специфического расширения вендора

Цель проще:

- сначала переносимый чат
- затем расширенные функции Claude
- отсутствие требования об обязательном форке

---

## 10. Критерии приемки

Эта инициатива считается завершенной, когда выполнены все следующие условия:

### Приемка продукта

- Пользователь может запустить Hermes Workspace со стандартным OpenAI-совместимым бэкендом и успешно общаться без внесения правок в код бэкенда.
- Пользователь может запустить Hermes Workspace с ванильным `hermes-agent` и получить работающий базовый опыт.
- Расширенные функции не приводят к критическому отказу приложения при отсутствии специфических API Claude.
- Интерфейс четко сообщает о переносимом режиме или расширенном режиме Claude.

### Техническая приемка

- Путь отправки чата больше не зависит жестко от `/api/sessions`.
- Проверка возможностей включает готовность `/v1/chat/completions`, а не только специфических API Claude.
- Отсутствие `/api/sessions`, `/api/skills`, `/api/memory` или `/api/config` не блокирует загрузку приложения или основной чат.
- Потоковая передача чата в переносимом режиме работает с OpenAI-совместимыми ответами SSE.

### Приемка документации

- README больше не говорит о необходимости форка.
- Документация по настройке сначала описывает OpenAI-совместимый автономный режим.
- Поддержка расширенного API Hermes Agent документирована как прогрессивное улучшение.
- Цель Шага 2 по переносу в основную ветку четко задокументирована.

---

## 11. Руководство по реализации

Это не детальный план задач, но инженерное направление должно быть следующим:

1. Разделить **клиент основного чата** и **клиент расширенного чата Claude**.
2. Рефакторинг проверки возможностей на уровни: переносимый vs расширенный.
3. Добавить путь парсера потока, совместимый с OpenAI.
4. Добавить откат к локальной ветке для бэкендов без поддержки сессий.
5. Чисто ограничить расширенные экраны проверками возможностей.
6. Переписать онбординг и документацию вокруг позиционирования "сначала переносимость".
7. После стабилизации Шага 1 подготовить PR в основную ветку (upstream) для нативных эндпоинтов Claude.

---

## 12. Итоговое заявление о решении

Закрепить следующее:

> Hermes Workspace — это автономный фронтенд для OpenAI-совместимых бэкендов чата.
>
> Нативные API Claude — это слой улучшений, а не требование.
>
> Шаг 1 — переносимая совместимость прямо сейчас.
>
> Шаг 2 — перенос расширенных API Hermes Agent в основную ветку, чтобы форк больше никогда не понадобился.
