# Спецификация фреймворка памяти Swarm2

Дата: 28.04.2026
Статус: Спецификация реализации
Канонический корень: `/Users/system/Documents/mybuild/hermes-workspace-rus`

## Цель

Воркеры Swarm2 нуждаются в долгосрочной памяти, которая выходит за рамки окна контекста одной сессии. Фреймворк памяти должен обеспечивать:

1. **Изоляцию по профилям**: каждый воркер владеет своей памятью.
2. **Контекст миссии**: общая память для воркеров, участвующих в одной миссии.
3. **Эпизодические логи**: хронологическая запись значимых событий и решений.
4. **Передачи (Handoffs)**: структурированные сводки для возобновления работы или передачи другому воркеру.
5. **Общие знания**: база знаний о проекте, доступная всему рою.
6. **Возможность поиска**: API для поиска по памяти во время планирования и исполнения.

## Структура хранения

Вся память воркеров должна храниться в директории профиля Hermes для обеспечения переносимости и изоляции.

### Корень памяти профиля

`~/.hermes/profiles/<workerId>/memory/`

### Файлы профиля

#### `MEMORY.md`

Курируемая долгосрочная память. Содержит факты, предпочтения пользователя и повторяющиеся выводы.

Шаблон:

```markdown
# Память — <workerId>

## Роль

<роль и специализация из swarm.yaml>

## Постоянные операционные заметки

- ...

## Соглашения проекта

- ...

## Извлеченные уроки

- ...
```

#### `IDENTITY.md`

Стабильные метаданные идентичности.

```markdown
# IDENTITY.md — <workerId>

- Имя: <отображаемое имя>
- ID воркера: <workerId>
- Роль: <role>
- Специализация: <specialty>
- Модель: <model>
```

#### `SOUL.md`

Инструкции роли/персоны для воркера.

Цель:

- специфическое для роли поведение,
- тон и стиль принятия решений,
- правила эскалации,
- планка качества.

### Память миссии

Путь:

`~/.hermes/profiles/<workerId>/memory/missions/<missionId>/`

Файлы:

- `SUMMARY.md` — человекочитаемый контекст миссии и текущее состояние
- `events.jsonl` — дополняемый поток структурированных событий
- `handoff.md` — последняя специфическая для миссии передача (если есть)

Шаблон `SUMMARY.md`:

```markdown
# Миссия <missionId> — <title>

## Текущее состояние

- Статус: планирование | выполнение | заблокировано | проверка | завершено
- Текущее задание: <assignmentId или none>
- Последнее обновление: <ISO timestamp>

## Цель

<цель миссии>

## Решения

- ...

## Затронутые файлы

- ...

## Контрольные точки

- ...

## Блокировщики

- ...

## Следующее действие

...
```

Формат события в `events.jsonl`:

```json
{"at":"2026-04-28T00:00:00.000Z","type":"dispatch","workerId":"swarm5","missionId":"mission-...","assignmentId":"assign-...","summary":"Отправлена задача строителю"}
```

Типы событий:

- `mission-start` (начало миссии)
- `dispatch` (диспетчеризация)
- `checkpoint` (контрольная точка)
- `handoff-requested` (запрос передачи)
- `handoff-written` (передача записана)
- `resume` (возобновление)
- `blocked` (заблокировано)
- `complete` (завершено)
- `note` (заметка)

### Эпизодические логи

Путь:

`~/.hermes/profiles/<workerId>/memory/episodes/YYYY-MM-DD.md`

Цель:

- хронологическая активность воркера,
- удобный поиск (grep),
- относительно "сырой", но читаемый вид.

Шаблон:

```markdown
# Эпизоды — <workerId> — YYYY-MM-DD

## HH:MM UTC — <тип события>

- Миссия: <missionId>
- Задание: <assignmentId>
- Сводка: ...
- Результат: ...
- Следующее действие: ...
```

### Передачи (Handoffs)

Локальная передача воркера:

`~/.hermes/profiles/<workerId>/memory/handoffs/<missionId>.md`

Общая последняя передача:

`/Users/system/.openclaw/workspace/memory/handoffs/swarm/<workerId>-latest.md`

Шаблон:

```markdown
# Передача — <workerId> — <missionId>

Сгенерировано: <ISO timestamp>

## Текущее состояние

...

## Цель

...

## Завершено

...

## В процессе

...

## Затронутые файлы

...

## Запущенные команды

...

## Блокировщики

...

## Следующее точное действие

...

## Промпт возобновления

При перезапуске этого воркера загрузите эту передачу, проверьте runtime.json, затем продолжите с раздела "Следующее точное действие".
```

## Контракты API

### `GET /api/swarm-memory`

Параметры запроса:

- `workerId` (обязателен, если не чтение общей памяти)
- `kind`: `profile | mission | episodic | handoff | shared`
- `missionId` (опционален/обязателен для памяти миссии)
- `date` (опционален для эпизодических логов)

Возвращает:

```json
{
  "ok": true,
  "workerId": "swarm5",
  "kind": "mission",
  "path": "...",
  "files": [
    { "name": "SUMMARY.md", "path": "...", "content": "..." }
  ]
}
```

### `POST /api/swarm-memory`

Тело запроса:

```json
{
  "workerId": "swarm5",
  "kind": "mission",
  "missionId": "mission-...",
  "eventType": "checkpoint",
  "content": "markdown или текст сводки",
  "event": { "state": "DONE", "result": "..." }
}
```

Правила:

- валидация `workerId`, `missionId` и путей (path traversal),
- создание директорий по требованию,
- атомарная запись markdown,
- атомарное добавление JSONL событий,
- опциональное дублирование передач в путь общей памяти.

### `GET /api/swarm-memory/search`

Параметры запроса:

- `workerId` (опционален),
- `query` (обязателен),
- `scope`: `worker | shared | all`, по умолчанию `worker`,
- `limit`, по умолчанию 10.

Реализация этапа 1:

- текстовый поиск типа grep по файлам markdown/jsonl,
- простое ранжирование: точная фраза > совпадение токенов > новизна,
- возврат сниппетов с путями и номерами строк.

Позже:

- использование нативного провайдера памяти Claude или внешнего провайдера для семантического поиска.

## Интеграция с существующими навыками

### `swarm-worker-core`

Уже определяет дисциплину контрольных точек и отчетность рантайма.

Фреймворк памяти добавляет:

- каждая контрольная точка может стать событием миссии,
- каждая контрольная точка может дополнить эпизодическую память,
- долгосрочные уроки могут быть перенесены в `MEMORY.md`.

### `swarm-dev-runtime`

Уже определяет инварианты профилей/tmux, нативные для Claude.

Фреймворк памяти должен соблюдать:

- корень профиля `~/.hermes/profiles/<workerId>`,
- `HERMES_HOME` для каждого воркера,
- tmux-сессии с именами `swarm-<workerId>`,
- обертки в `~/.local/bin/swarmN`.

### `swarm-orchestrator`

Использует память для:

- поиска в истории воркера перед назначением задачи,
- прикрепления релевантного контекста миссии к промптам,
- поиска предыдущих передач,
- архивации завершенных миссий.

### Навыки ролей

`swarm-pr-worker`, `swarm-ui-worker`, `swarm-bench-worker` и будущие навыки ролей используют `swarm-memory` для специфического поиска по ролям.

### `self-improving-agent`

Обеспечивает соглашения об обучении и продвижении знаний.

Фреймворк памяти принимает:

- паттерн ежедневных логов,
- `MEMORY.md` как курируемую долгосрочную память,
- продвижение широко полезных знаний.

## Поведение при запуске/возобновлении

Будущий навык `swarm-memory` должен инструктировать воркеров загружать:

1. `IDENTITY.md`, `SOUL.md` и `MEMORY.md`,
2. `runtime.json` для определения `currentMissionId`,
3. активную миссию `SUMMARY.md`, если есть,
4. последнюю локальную/общую передачу для текущей миссии, если есть,
5. недавние эпизодические записи, относящиеся к активной миссии.

Промпт возобновления должен быть компактным:

```text
Загрузи память воркера из ~/.hermes/profiles/<workerId>/memory/.
Если в runtime.json есть currentMissionId, прочитай SUMMARY.md этой миссии и последнюю передачу (handoff).
Продолжи с раздела передачи "Следующее точное действие".
```

## Хуки автоматической записи

Хуки этапа 1:

- начало диспетчеризации → событие миссии + эпизодическая запись,
- распарсена контрольная точка → событие миссии + эпизодическая запись + обновление SUMMARY.md,
- запрошена передача жизненного цикла → событие `handoff-requested`,
- записана передача → локальная передача + общая последняя передача.

## Безопасность и приватность

- Не копировать приватную `MEMORY.md` основной сессии в профили воркеров.
- Воркеры получают только свою профильную память плюс память миссии/общую память роя.
- Общая память роя должна содержать знания о проекте/процессе, а не приватный контекст пользователя, если это не предусмотрено явно.
- Все записи должны оставаться внутри канонических корней.
