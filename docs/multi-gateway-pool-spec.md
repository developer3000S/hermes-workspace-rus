# Архитектура пула шлюзов (Multi-Gateway Pool)
## Hermes Workspace — параллельное выполнение агентов по профилям

### Статус: Проектный документ — предложение к PR

---

## 1. Постановка проблемы

В настоящее время Hermes Workspace работает как **интерфейс для одного шлюза и одного профиля**. Шлюз загружает один `HERMES_HOME` при запуске, и все чат-сессии, операции и доступ к памяти проходят через этот единственный процесс.

Для пользователей с несколькими профилями (основной сценарий использования Claude) это означает:
- **Отсутствие параллельного выполнения агентов**: нельзя проводить мозговой штурм с Nous, пока Jules оркеструет Architect и Sentinel в разделе "Операции".
- **Отсутствие идентичности профиля в чате**: "агентом" всегда является тот, под чьим именем был запущен единственный шлюз.
- **Фрагментация терминала**: пользователям приходится открывать отдельные окна терминала для каждого профиля, чтобы добиться по-настоящему мультиагентных рабочих процессов.
- **Засорение сессий**: все сессии сваливаются в одну кучу, независимо от того, какой агент их создал.

### Пользовательская история

> "Я хочу стратегически мыслить вместе с Nous о функциях Ascent Performance, в то время как Jules оркеструет сборку с помощью Architect и Sentinel. Я хочу быстро переключаться между этими диалогами с агентами в одном окне рабочего пространства, при этом каждый агент должен сохранять свою память, навыки и контекст сессии."

## 2. Принципы проектирования

**Фундаментальная истина**: каждый профиль Hermes — это **отдельный когнитивный агент** с собственным SOUL.md, навыками, памятью и целью. Это не просто "режимы" одного агента. Это параллельно работающие агенты.

**Следствие**: рабочее пространство должно быть **оркестратором агентов**, а не просто визуальной оболочкой над одним шлюзом.

**Ограничение**: шлюз Hermes Agent спроектирован как однопользовательский процесс. Он не может динамически перезагружать профили. Каждому профилю требуется собственный экземпляр шлюза.

**Решение**: рабочее пространство поддерживает **пул шлюзов** — по одному процессу шлюза на каждый активный профиль, каждый на своем порту, с мониторингом состояния и возможностью маршрутизации из UI.

**Принципы проектирования:**
1. **Независимость от количества профилей**: работает как для 1 профиля, так и для 100. Никаких жестко закодированных лимитов, массивов или условий перечисления конкретных профилей.
2. **Приватность по умолчанию**: отсутствие PII, API-ключей, паролей или секретов в коде, логах или PR. Все конфиденциальные данные остаются в локальных `.env` файлах профилей.
3. **Обратная совместимость**: пользователи с одним профилем не затронуты. Режим пула является опциональным.
4. **Отказоустойчивость**: неработающий шлюз не приводит к краху всего рабочего пространства. Всегда доступен безопасный откат к базовому состоянию.

## 4. Менеджер пула шлюзов

### 4.1 Соглашение о назначении портов

```typescript
const BASE_PORT = 8642
function getGatewayPort(profileName: string, profiles: string[]): number {
  const index = profiles.indexOf(profileName)
  return BASE_PORT + Math.max(0, index)
}
```

Профили сортируются по алфавиту для обеспечения стабильного назначения портов. Файл состояния (`gateway-pool.json`) запоминает назначения между перезапусками.

**Независимость от количества профилей**: менеджер пула динамически обнаруживает профили в файловой системе (`~/.hermes/profiles/*`). Нет жестко заданного списка, максимального количества или специальных условий для конкретных имен профилей. Код для пользователя с 2 профилями и для пользователя с 50 профилями идентичен.

### 4.2 Состояния жизненного цикла шлюза

```typescript
type GatewayState = 
  | 'spawning'      // Процесс запускается
  | 'healthy'       // Ответил на /health в течение 5с
  | 'degraded'      // Медленные ответы (>2с)
  | 'dead'          // Ошибка проверки здоровья 3 раза подряд
  | 'stopped'       // Явно остановлен пользователем
```

### 4.3 Протокол запуска (Spawn)

```typescript
function spawnGateway(profileName: string, port: number): ChildProcess {
  const profilePath = path.join(getClaudeRoot(), 'profiles', profileName)
  const env = {
    ...process.env,
    HERMES_HOME: profilePath,
    CLAUDE_GATEWAY_PORT: String(port),
    CLAUDE_PROFILE_NAME: profileName,
  }
  return spawn('claude', ['gateway', '--port', String(port)], { env })
}
```

**Примечание**: шлюз должен запускаться через команду `hermes gateway`, а не через внутренний `gateway.ts` рабочего пространства. Workspace становится оркестратором, а не хостом шлюза.

### 4.4 Монитор здоровья

- Опрос `GET /health` каждого шлюза каждые 30с.
- 3 последовательных сбоя → пометить как `dead`, автоперезапуск (с экспоненциальной задержкой).
- Медленный ответ (>2с) → пометить как `degraded`, записать предупреждение в лог.
- Восстановление → пометить как `healthy`.

### 4.5 Протокол завершения работы

При выходе из рабочего пространства (SIGTERM):
1. Отправить сигнал плавного завершения всем шлюзам (`POST /shutdown`).
2. Подождать 10с.
3. Принудительно завершить (SIGKILL) оставшиеся.
4. Сохранить состояние в `gateway-pool.json`.

## 5. Слой маршрутизации запросов

### 5.1 Изменения в маршрутах API

Все API-маршруты рабочего пространства получают **контекст профиля**:

```typescript
// Сейчас: /api/chat/completions
// Будет:  /api/chat/completions?profile=nous
//         или заголовок: X-Claude-Profile: nous

// Прокси-маршруты к шлюзам:
// /api/gateway/{profile}/chat/completions
// /api/gateway/{profile}/sessions
// /api/gateway/{profile}/memory
// и т.д.
```

### 5.2 Реализация роутера

```typescript
// src/server/gateway-router.ts
export async function proxyToGateway(
  profileName: string,
  path: string,
  init?: RequestInit
): Promise<Response> {
  const gateway = gatewayPool.get(profileName)
  if (!gateway || gateway.state !== 'healthy') {
    throw new Error(`Шлюз для профиля "${profileName}" недоступен`)
  }
  const url = `http://127.0.0.1:${gateway.port}${path}`
  return fetch(url, init)
}
```

### 5.3 Обратная совместимость

Если профиль не указан:
- Использовать `activeProfile` по умолчанию (из файла `~/.hermes/active_profile`).
- Если файл не существует, использовать первый доступный профиль.
- Пользователи с одним профилем не заметят **никаких изменений в поведении**.

## 6. Модель изоляции сессий

### 6.1 Хранение сессий

Сейчас: все сессии в `~/.hermes/sessions/` (или в папке сессий профиля).

С мульти-шлюзом: каждый шлюз управляет своими сессиями в собственной директории профиля. Рабочее пространство **агрегирует** их для отображения, но **маршрутизирует** их по профилям.

```typescript
// src/server/sessions-aggregator.ts
export async function listAllSessions(): Promise<SessionMeta[]> {
  const profiles = listProfiles()
  const allSessions = await Promise.all(
    profiles.map(async (profile) => {
      const gateway = gatewayPool.get(profile.name)
      if (!gateway) return []
      const res = await fetch(`http://127.0.0.1:${gateway.port}/api/sessions`)
      const sessions = await res.json()
      return sessions.map((s: SessionMeta) => ({
        ...s,
        profile: profile.name,
        profileColor: getProfileColor(profile.name),
      }))
    })
  )
  return allSessions.flat().sort((a, b) => b.updatedAt - a.updatedAt)
}
```

### 6.2 Отображение сессий

Сессии в боковой панели **сгруппированы по профилям** с визуальным различием:

```
СЕССИИ
▼ nous (зеленая точка)
  ├─ Привет из теста workspace · 15:05
  └─ Архитектура сортировки почты · 22 апр.
▼ jules (синяя точка)
  ├─ Оркестрация сборки Ascent · 14:30
  └─ Координация PR #118 · 21 апр.
▼ architect (фиолетовая точка)
```

## 7. Изменения в интерфейсе (UI)

### 7.1 Селектор профиля (Глобальный заголовок)

Постоянная кнопка в верхнем левом углу (рядом с переключателем боковой панели):

```
[☰] [ nous ▼ ]              Hermes Workspace
```

- Выпадающий список всех профилей с индикаторами статуса.
- Зеленая точка = шлюз здоров.
- Желтая точка = деградация.
- Красная точка = мертв/остановлен.
- Клик переключает активный профиль для **текущей панели**.
- Горячие клавиши `Cmd+Shift+1..6` для быстрого переключения.

### 7.2 Экран чата

- Пустое состояние показывает имя активного профиля и модель.
- Список сессий показывает точки цветов профилей для каждой сессии.
- Область ввода отправляет сообщения на шлюз активного профиля.
- "Новая сессия" создает сессию в рамках активного профиля.

### 7.3 Операции / Conductor

- Карточки задач показывают, на каком профиле они запущены.
- Выпадающий список "Запустить на" при создании задач.
- Дашборд операций агрегирует задачи по всем профилям.

### 7.4 Браузер памяти

- Записи памяти помечены профилем.
- Фильтрация по профилю.
- Поиск по памяти нескольких профилей (опционально, настраивается пользователем).

## 10. Обработка ошибок и пограничные случаи

| Сценарий | Поведение |
|----------|----------|
| Ошибка запуска шлюза | Показать уведомление, разрешить повтор, откат к активному профилю |
| Порт уже занят | Автоматическое увеличение номера порта, запись предупреждения в лог |
| Профиль удален при работающем шлюзе | Остановить шлюз, удалить из пула |
| Крах Workspace | При перезапуске проверить наличие осиротевших шлюзов, подхватить или убить |
| Пользователь с одним профилем | Режим пула выключен по умолчанию, влияние нулевое |
| Несоответствие версий шлюзов | Записать предупреждение, все равно попытаться запустить |
| Нехватка памяти | Позволить пользователю останавливать простаивающие шлюзы |

## 11. Безопасность и приватность

- Шлюзы привязываются только к `127.0.0.1` (уже по умолчанию).
- Отсутствие утечек памяти между профилями (у каждого шлюза свой `HERMES_HOME`).
- Селектор профилей учитывает авторизацию.
- Доступ к запуску/остановке шлюзов только для админа.
- **Никаких секретов в коде или PR**: API-ключи, пароли, токены и PII никогда не должны появляться в исходном коде, тестах, логах или описаниях PR. Вся чувствительная конфигурация живет в локальных `.env` файлах профилей, которые добавлены в `.gitignore`.
- **Обезличенные примеры**: диаграммы архитектуры и примеры используют вымышленные имена профилей (например, `agent-alpha`, `agent-beta`) или общие заглушки.
- **Никаких жестко заданных путей**: назначения портов, директории профилей и URL шлюзов вычисляются динамически. Никаких путей вроде `/Users/...` в коде.
- **Безопасность логов**: логи пула шлюзов должны скрывать любые переменные окружения, содержащие `KEY`, `TOKEN`, `SECRET` или `PASSWORD`.

## 13. Обратная совместимость

- **Пользователи с одним профилем**: никак не затронуты. Режим пула выключен по умолчанию.
- **Текущие мульти-профильные пользователи**: режим пула можно включить в настройках. Когда он выключен, поведение соответствует текущему режиму одного шлюза.
- **Существующие сессии**: сохраняются. Каждая сессия уже живет в папке своего профиля. Workspace просто будет корректно их агрегировать.
- **Контракты API**: все существующие маршруты `/api/*` работают без изменений, если профиль не указан.
