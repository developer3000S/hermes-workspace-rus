# Спецификация IDE агентов Swarm2

Статус: Черновик v1
Дата: 27.04.2026
Владелец: Aurora / Eric
Интерфейс: `/swarm2`, плюс общие панели агентов в основном чате рабочего пространства

## 1. Тезис продукта

Swarm2 — это не дашборд. Swarm2 — это плоскость управления и IDE для **суб-агентов Claude**.

Пользователь должен иметь возможность клонировать своего основного агента Claude столько раз, сколько потребуется. Каждый клон обладает тем же уровнем доступа, структурой контекста, навыками, правилами памяти, инструментами рабочего пространства и проектными возможностями, что и родительский агент, но может работать независимо над своей линией задач.

Это существенно отличается от запуска одноразовых субагентов:

- Обычные субагенты — это эфемерные исполнители.
- Суб-агенты Claude — это постоянные клонированные операторы.
- Обычные субагенты выполняют конкретную задачу.
- Суб-агенты Claude владеют целой линией/проектом и могут работать на автопилоте.
- Обычные субагенты теряют преемственность, если не обернуты специальным образом.
- Суб-агенты Claude несут в себе постоянное состояние профиля, метаданные среды выполнения, задачи, логи, сессии, навыки и понимание проекта.

Обещание продукта Swarm:

> Клонируйте своего агента в команду, дайте каждому клону свою линию работы, наблюдайте за процессом в реальном времени, вмешивайтесь при необходимости и инспектируйте/редактируйте/отправляйте результаты, не покидая IDE Swarm.

## 2. Основные концепции

### 2.1 Главный агент / Хаб (Hub)

Главный агент является оркестратором.

В `/swarm2` главный агент должен отображаться как компактный узел/карточка хаба, а не как гигантская панель чата. Его работа заключается в следующем:

- маршрутизация задач
- мониторинг агентов
- координация совместной работы
- вынос на поверхность решений и блокировщиков
- поддержание топологии роя (swarm)

Карточка хаба должна показывать:

- аватар / личность
- количество активных участников роя
- количество комнат / совместных работ
- блокировщики / ошибки авторизации
- активные проектные линии
- элементы управления маршрутизацией

Хаб не должен занимать основную область контента полноценным интерфейсом чата.

### 2.2 Суб-агент Hermes

Агент роя — это клонированный профиль Hermes с:

- конфигурацией модели/провайдера
- доступом к навыкам/инструментам
- файлами памяти/профиля
- метаданными среды выполнения
- сессиями/историей
- состоянием задач/cron/автопилота
- рабочей директорией проекта (cwd)
- логами/выводом
- сессией терминала/TUI
- опциональным сервером разработки / URL для предпросмотра

Каждая карточка воркера (рабочего агента) должна быть компактной плиткой IDE для этого клона.

### 2.3 Линия агента (Lane)

Каждый клонированный агент может владеть своей линией:

- проектная линия, например BenchLoop
- линия PR (запросов на слияние)
- линия исследований
- линия проверки (ревью)
- линия очереди задач (issue queue)
- линия документации
- линия операций (ops)

Линия включает:

- текущую цель
- очередь задач
- контекст проекта
- живой чат
- терминал/TUI
- поверхности предпросмотра/редактора
- состояние передачи/контрольной точки

### 2.4 Автопилот

Автопилот — это оркестрация над постоянными клонированными агентами.

Автопилот должен:

- назначать задачи подходящему клонированному агенту
- поддерживать движение агентов с помощью контрольных точек на основе доказательств (proof-based checkpoints)
- обнаруживать отклонения/застои/блокировщики
- перезапрашивать (re-prompt) или перемаршрутизировать задачи при необходимости
- суммировать значимые результаты
- избегать прерывания пользователя, если только не требуется решение, возник блокировщик или получен конечный результат

## 3. Необходимые интерфейсы

### 3.1 Хаб роя (Swarm Hub)

Компактная карточка хаба в верхней центральной части.

Необходимо:

- Личность/аватар главного агента
- Метрики статуса роя
- Кнопка маршрутизации / диспетчеризации
- Количество активных комнат/коллабораций
- Состояние блокировщиков/авторизации
- Связи с карточками воркеров

Не требуется:

- Полноценная встроенная панель основного чата

### 3.2 Карточки воркеров (Worker Cards)

Каждая карточка воркера является основным узлом управления.

Необходимые поля:

- имя/id воркера
- роль/линия
- статус: ожидает, работает, думает, проверяет, заблокирован, ждет ответа, оффлайн
- модель/провайдер
- текущая задача
- предпросмотр очереди задач
- последние сообщения чата
- последние сигналы среды выполнения/логов
- проект/директория/ветка
- URL предпросмотра, если доступен
- состояние подключения терминала/TUI
- быстрые действия: чат, маршрут, задачи, терминал, предпросмотр, редактор

### 3.3 Живой чат воркера

Чат воркера должен быть настоящим, а не просто локальной историей карточки.

Необходимо:

- Получение последних сообщений из реальной истории сессий/state.db/API воркера
- Опрос или подписка на обновления
- Компактное отображение сообщений пользователя/ассистента/инструментов
- Отправка из карточки направляется в реальный профиль/сессию этого воркера
- LocalStorage разрешен только как запасной вариант или кэш черновиков

Критерии приемки:

- Если воркер получил промпт от автопилота, карточка отражает этот диалог.
- Если пользователь отправляет сообщение воркеру, ответ появляется в той же живой ленте.

### 3.4 Очередь задач

Каждая карточка воркера должна показывать состояние своих задач.

Необходимо:

- текущая задача
- следующие 2-3 задачи в очереди
- индикаторы блокировки/ожидания
- количество выполненных/недавно завершенных задач
- клик для открытия полной панели задач или переход на `/tasks?assignee=workerId`

Источники задач:

- API задач Hermes
- текущая задача/состояние из runtime.json воркера
- опционально метаданные issue/PR проекта

### 3.5 Живой Терминал / TUI

Swarm2 должен хостить актуальный живой терминал/TUI воркера, если он доступен.

Необходимо:

- автоопределение активных терминалов воркеров
- монтирование активных терминалов по умолчанию в представлении рантайма
- возможность добавления/удаления панелей терминала
- подключение к реальному tmux/сессии, когда доступно
- откат к живой ленте логов или оболочке, если подключаемый терминал отсутствует
- сохранение точного взаимодействия с TUI, где это возможно

Критерии приемки:

- Пользователь может использовать TUI агента напрямую из Swarm2.
- Представление рантайма не должно требовать ручного ввода имени сессии tmux.

### 3.6 Предпросмотр проекта

Если агент работает над локальным проектом/сайтом, Swarm2 должен показывать его.

Необходимо:

- определение рабочей директории проекта
- определение конфигурации пакета/сервера разработки, где это возможно
- отображение известных локальных URL (например, порты localhost)
- встроенная панель предпросмотра
- отображение названия проекта, ветки, измененных файлов

Критерии приемки:

- Если swarm4 работает над BenchLoop и запущен локальный предпросмотр, пользователь может видеть его внутри Swarm2, не покидая страницу.

### 3.7 Редактор / Поверхность Diff

Swarm2 должен эволюционировать в IDE для агентов, подобную Bolt.

Этап 1:

- браузер файлов для рабочей директории воркера
- открытие/чтение файлов
- редактирование/сохранение файлов
- просмотр диффов (различий)
- список измененных файлов

Этап 2:

- визуальный выбор элементов из предпросмотра
- выбранный элемент -> создание задачи на исправление
- правки с помощью агента
- рабочий процесс принятия/отклонения патчей

Критерии приемки:

- Пользователь может инспектировать и изменять вывод агента прямо из интерфейса роя.

### 3.8 Представление "Офис" / Коллаборация

Карточки должны визуализировать совместную работу.

Концепция:

- у каждого воркера есть небольшое офисное пространство 2x2 в его карточке
- инструменты/статус подсвечивают области офиса
- когда воркеры сотрудничают, их карточки/офисы соединяются в большой общий офис
- линии (провода) показывают пути коллаборации и маршрутизацию

Необходимо в первой итерации:

- визуальная группировка по проектам/линиям
- усиленная индикация связи для воркеров в одной комнате/проекте
- акцент на линиях коллаборации

## 4. Схема реализации

### 4.1 Бэкенд (Backend)

- `/api/swarm-roster`: чтение списка воркеров и их метаданных.
- `/api/swarm-runtime`: агрегация состояния воркеров из `runtime.json`, tmux и API заданий.
- `/api/swarm-chat`: получение последних сообщений из `state.db` каждого профиля.
- `/api/swarm-dispatch`: отправка сообщений/задач конкретному воркеру.
- `/api/swarm-control`: запуск/остановка сессий воркеров.

### 4.2 Фронтенд (Frontend)

- `Swarm2Screen`: основной контейнер.
- `SwarmHub`: компонент центрального хаба.
- `WorkerCard`: основной блок воркера.
- `WorkerChatOverlay`: живой чат внутри/поверх карточки.
- `WorkerTerminalView`: встроенный TUI/логи.
- `WorkerTaskOverlay`: управление очередью.
- `WorkerPreviewOverlay`: iframe предпросмотра проекта.
- `WorkerEditorOverlay`: поверхность редактирования файлов и диффов.

## 5. Этапы развития

### Этап 1: Живая плоскость управления

- Карточка хаба для Aurora.
- Карточки воркеров с живым статусом и текущей задачей.
- Живой чат воркера (только чтение/получение).
- Кнопка маршрутизации для отправки промптов воркерам.
- Группировка по проектам.

### Этап 2: Рантайм и Задачи

- Встроенный TUI/Терминал в карточке.
- Управление очередью задач в карточке.
- Логи рантайма.
- Автоматическая прокрутка и фокус на активном воркере.

### Этап 3: IDE и Предпросмотр

- Встроенный предпросмотр проекта.
- Браузер файлов и редактор.
- Просмотр диффов и принятие изменений.
- Глубокая интеграция "выбери и исправь".
