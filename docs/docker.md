# Docker

Hermes Workspace + Hermes Agent в контейнерах.

## Краткое руководство (один хост, только localhost)

```bash
git clone https://github.com/outsourc-e/hermes-workspace
cd hermes-workspace
cp .env.example .env
# добавьте хотя бы один ключ провайдера (например, OPENROUTER_API_KEY=...)
docker compose up -d
open http://localhost:3000
```

Это всё. `docker-compose.yml` из репозитория запускает:

- `hermes-agent` (порт `8642`, только внутренний)
- `hermes-workspace` (порт `3000`, привязан к `127.0.0.1`)

Рабочее пространство ожидает, пока `/health` агента вернет `200`, прежде чем запуститься (через `depends_on: condition: service_healthy`). На новом ноутбуке это занимает около 15 секунд.

## Несколько хостов / NAS / VPS

Если рабочее пространство и агент работают на **разных машинах**, или вы хотите получить доступ к рабочему пространству через локальную сеть / Tailscale, меняются три вещи:

### 1. Агент привязывается к публичному адресу

В `.env`:

```bash
API_SERVER_HOST=0.0.0.0
API_SERVER_KEY=<длинная случайная строка>
```

Это заставляет агента слушать все интерфейсы, а не только петлю (loopback) Docker. **`API_SERVER_KEY` обязателен**, когда `API_SERVER_HOST` не является loopback — в противном случае агент откажется запускаться.

### 2. Workspace знает, где находится агент

В `.env`:

```bash
HERMES_API_URL=http://<хост-или-сервис-агента>:8642
HERMES_API_TOKEN=<то же значение, что и у API_SERVER_KEY>
HERMES_DASHBOARD_URL=http://<хост-или-сервис-агента>:9119
HERMES_DASHBOARD_TOKEN=<тот же ключ или установите CLAUDE_DASHBOARD_TOKEN>
```

Внутри docker compose на том же хосте `<хост-или-сервис-агента>` — это имя сервиса из вашего файла compose (например, `hermes-agent`). На Synology NAS с отдельным стеком рабочего пространства — это IP-адрес локальной сети (например, `192.168.1.78`).

### 3. Workspace получает пароль

В Docker Workspace привязывается не к loopback (`0.0.0.0:3000`). Он отказывается запускаться в режиме продакшена без пароля, чтобы предотвратить случайный открытый доступ:

```bash
HERMES_PASSWORD=<длинная случайная строка, отличная от API_SERVER_KEY>
```

Если вы публикуете рабочее пространство через HTTPS (обратный прокси, Tailscale Funnel, Cloudflare Tunnel), также установите `COOKIE_SECURE=1`, чтобы куки сессии получили флаг `Secure`.

## Сбои подключения — руководство по диагностике

Если рабочее пространство показывает "**Disconnected**" или "**Missing Hermes APIs detected**", но агент кажется запущенным:

### Шаг 1 — Проверьте доступность агента из контейнера workspace

```bash
docker compose exec hermes-workspace sh
# внутри контейнера workspace:
curl -fsS http://hermes-agent:8642/health
curl -fsS -H "Authorization: Bearer $HERMES_API_TOKEN" http://hermes-agent:8642/v1/models | head -c 200
exit
```

Если `/health` возвращает JSON `{"status": "ok"}`, агент жив в сети docker.

### Шаг 2 — Проверьте окружение workspace

```bash
docker compose exec hermes-workspace env | grep -E "HERMES_API|API_SERVER"
```

Вы должны увидеть:

- `HERMES_API_URL=http://hermes-agent:8642` (или имя соответствующего сервиса)
- `HERMES_API_TOKEN=<то же значение, что и API_SERVER_KEY агента>`

### Шаг 3 — Принудительно запустите повторную проверку

Рабочее пространство кэширует карту возможностей шлюза на 2 минуты (на 15 секунд в состоянии отключения, начиная с версии v2.2.1). Если агент запустился после того, как рабочее пространство начало проверку, этот кэш устарел.

```bash
curl -X POST http://localhost:3000/api/gateway-reprobe
```

Это перезапустит проверку и вернет свежую карту возможностей. Если теперь там указано `mode=zero-fork`, значит вы подключены.

### Шаг 4 — Изучите лог возможностей рабочего пространства

Workspace логирует полную сводку возможностей при каждой проверке. Ищите строку `[gateway]`:

```bash
docker compose logs hermes-workspace 2>&1 | grep '\[gateway\]' | tail -3
```

Здоровый лог выглядит так:

```
[gateway] gateway=http://hermes-agent:8642 dashboard=http://hermes-agent:9119 mode=zero-fork core=[health,chatCompletions,models,streaming] enhanced=[sessions,skills,memory,config,jobs,enhancedChat,conductor,kanban] missing=[mcp]
```

Лог с ошибкой обычно показывает `core=[]` и `missing=[health,...]` — это означает, что каждая проверка получила ответ, отличный от 2xx. Проверьте логи агента (`docker compose logs hermes-agent`) на наличие соответствующих записей 401/404/timeout.

### Распространенные причины

| Симптом | Причина | Исправление |
|---|---|---|
| `core=[]` и `missing=[health,...]` | Workspace проверил агента до того, как тот был готов | Подождите 30 сек и обновите страницу, или выполните `POST /api/gateway-reprobe`. TTL кэша падает до 15 сек в состоянии отключения. |
| `core=[health,chatCompletions]`, но нет `models` | Старый образ агента (до появления `/v1/models`) | Обновитесь: `docker compose pull && docker compose up -d` |
| Все проверки возвращают 401 | `HERMES_API_TOKEN` не совпадает с `API_SERVER_KEY` агента | Убедитесь, что оба значения в `.env` одинаковы. Они должны совпадать в точности. |
| UI рабочего пространства показывает "Connection refused" | Workspace использует `127.0.0.1` вместо имени сервиса | Установите `HERMES_API_URL=http://hermes-agent:8642` (или имя соответствующего сервиса). |
| Агент циклически перезагружается с ошибкой `API_SERVER_KEY required` | Агент привязан к 0.0.0.0 без ключа | Установите `API_SERVER_KEY` в `.env` (обязательно для привязки не к loopback). |

## Настройка Synology NAS / внешних хостов

Если ваше рабочее пространство и агент находятся в **разных стеках** на одном и том же NAS (или вообще на разных хостах), они не делят общую сеть docker. Вам необходимо:

1. Опубликовать порты обоих (агент на `8642`, рабочее пространство на `3000`).
2. Указать рабочему пространству на **хостовый IP** агента, а не на имя сервиса. Пример для Synology с NAS по адресу `192.168.1.78`:

```bash
HERMES_API_URL=http://192.168.1.78:8642
HERMES_API_TOKEN=<API_SERVER_KEY>
HERMES_DASHBOARD_URL=http://192.168.1.78:9119
```

3. Привязать агент к `0.0.0.0`:

```bash
API_SERVER_HOST=0.0.0.0
API_SERVER_KEY=<длинная случайная строка>
```

4. Плагину дашборда (мульти-доски канбан, миссии проводника) требуется запущенный сервис дашборда на хосте агента — см. docker-compose агента для этого сервиса.

Если вы привяжете агент к `0.0.0.0` на NAS без `API_SERVER_KEY`, агент откажется запускаться. Это сделано намеренно — выставление эндпоинта чата агента в открытый интернет без авторизации было бы выстрелом в ногу.

## Hermes Workspace + Hermes Agent: зачем два контейнера?

Workspace — это **интерфейс (UI)**. Agent — это **движок**. Разделение позволяет вам:

- Обновлять их независимо (`docker compose pull hermes-workspace` и т.д.)
- Запускать несколько рабочих пространств для одного агента (на разных портах)
- Запускать рабочее пространство на планшете/телефоне, пока агент работает на мощной машине

Стандартный файл compose размещает их вместе для простоты. Настройка с разделением хостов выше — это путь для тех, кто "понимает, что делает".

## Сообщения об ошибках

Если ваша настройка соответствует описанному выше руководству, но всё равно не работает, создайте issue на странице <https://github.com/outsourc-e/hermes-workspace/issues>, приложив:

1. Ваш `docker-compose.yml` (скройте секреты)
2. Вывод команды `docker compose logs hermes-workspace 2>&1 | grep '\[gateway\]' | tail -5`
3. Вывод команды `curl -fsS http://<хост-рабочего-пространства>:3000/api/gateway-reprobe -X POST` (также скройте секреты)

Это позволит нам найти истинную причину за пару комментариев вместо долгой переписки.
