## Что было реализовано в этой сессии

### 1. Лендинг HermesWorld переработан в автономную публичную страницу
Маршруты в репозитории:
- `/hermes-world`
- `/world`

Ключевой файл:
- `src/screens/playground/hermes-world-landing.tsx`

Теперь у лендинга есть:
- Публичная структура для запуска продукта
- Главный экран (hero), полоса возможностей, раздел с анонсом запуска, зоны, агенты, сигилы, финальный призыв к действию (CTA), футер
- Ссылки на GitHub / дорожную карту / список функций вместо ссылок на локальный `/playground`
- Более специфичные тексты для HermesWorld
- Сообщения к дню запуска, такие как:
  - “Запускаем HermesWorld сегодня.”
  - Страница лендинга / дорожная карта / список функций / спринт по графике

### 2. Лендинг HermesWorld теперь обходит оболочку рабочего пространства (workspace chrome)
Самым большим источником путаницы была утечка корневой оболочки приложения.

Исправлено или частично исправлено:
- `/hermes-world` и `/world` теперь обходят **WorkspaceShell** и прокручиваются как обычные страницы
- Из маршрута лендинга удалена локальная оболочка рабочего пространства
- Сплеш-скрин приложения пропускается на маршрутах лендинга HermesWorld
- Маршрут предназначен для прокрутки документа, а не для прокрутки панели приложения

Затронутые файлы:
- `src/components/workspace-shell.tsx`
- `src/routes/__root.tsx`

### 3. Отслежены и удалены/ограничены многочисленные оверлеи корневого уровня
Обнаружены и устранены следующие источники утечек на корневом уровне:
- `UsageMeter`
- `SearchModal`
- `KeyboardShortcutsModal`
- `UpdateCenterNotifier`
- `MobilePromptTrigger`
- `OnboardingTour`

Важное примечание:
- Некоторые из них были ограничены только для маршрутов лендинга.
- Позже игровые поверхности, такие как `/playground`, также потребовали специальной обработки.

### 4. Устранена утечка плашки использования/сессии в нижнем правом углу
Этот виджет `SESSION | IN | OUT | CTX | COST` был еще одной утечкой оверлея рабочего пространства.

Предпринятые действия:
- Глобальная плашка использования/сессии скрыта для **игровых поверхностей**, а не только для лендинга

Затронутые файлы:
- `src/routes/__root.tsx`

### 5. Устранены ошибки наложения боковой панели в `/playground`
На порту 3002 при открытой левой боковой панели элементы игрового интерфейса сильно перекрывались.

Реализованное направление исправления:
- Сделано так, чтобы несколько элементов игрового интерфейса учитывали боковую панель и смещались при её открытии

Затронутые файлы:
- `src/screens/playground/components/playground-hud.tsx`
- `src/screens/playground/components/playground-chat.tsx`
- `src/screens/playground/playground-screen.tsx`

Намеченное исправление:
- использовать `useWorkspaceStore((s) => s.sidebarCollapsed)`
- смещать HUD/чат/чипы на `320px` влево при открытой панели
- сохранять поведение `min(120px, 9vw)` при свернутой панели

### 6. Главный экран (hero) лендинга HermesWorld возвращен к стилистике "светящейся сферы"
Эрик предпочел старую атмосферу входа через сферу/портал вместо нового варианта с множеством скриншотов в верхней части.

Предпринятые действия:
- Главный экран переключен на использование `PlaygroundHeroCanvas`
- Сохранена новая структура/тексты лендинга, но визуальный стиль возвращен к энергии свечения сферы / портала / запуска

Затронутые файлы:
- `src/screens/playground/hermes-world-landing.tsx`
- `src/screens/playground/components/playground-hero-canvas.tsx` (используется, но не обязательно изменен)

### 7. Очистка текстов HermesWorld
Удалено больше остаточного текста рабочего пространства с лендинга.

Примеры изменений:
- `Hermes Workspace Experiment // Persistent Agent World`
  → `HermesWorld Preview // Persistent Agent World`
- `Hermes Workspace Connected`
  → `Live World Build`

### 8. Отладка локального превью/процесса разработки
Первопричина многих странностей:
- Несколько серверов Vite dev конфликтовали из-за `routeTree.gen.ts`
- Это вызывало чехарду с перезагрузкой, прерывание запросов и ощущение "остановки на полуслове" во время локального превью

Ключевой факт:
- Нестабильность 3002/3003 была связана не только с тем, что "сайт сломан", но и с нестабильностью локального процесса разработки
- В некоторые моменты использовались перезапуски через `detached/nohup`, чтобы поддерживать работу дольше
