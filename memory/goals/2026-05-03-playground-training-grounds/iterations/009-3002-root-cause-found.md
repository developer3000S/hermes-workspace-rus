# Итерация 009 — ОБНАРУЖЕНА ПЕРВОПРИЧИНА циклической загрузки на 3002

Дата: 2026-05-04 01:27 EDT
Репозиторий: `/Users/aurora/hermes-workspace`
Ветка: `main` (head: `72dd321e8`)

---

## Первопричина

**Три одновременных процесса `vite dev` работали с одним и тем же исходным деревом `hermes-workspace`.**

Список процессов на начало сессии:
- pid 40934 — vite dev на **:3001** (работает 36+ мин, рабочая директория `/Users/aurora/hermes-workspace`)
- pid 8826  — vite dev на **:3003** (работает 12+ часов, та же директория)
- pid 8866  — vite dev (зомби-пара для 8826, та же директория)
- pid 50028 — vite dev на **:3002** (тот, который нас интересовал, та же директория)
- pid 71649 — vite dev на **:3005** (другая директория: `/Users/aurora/.worktrees/hermes-playground-local`) — **нормально**
- pid 11433 — vite dev на **:3006** (worktree, нормально)

Все три дублирующих сервера vite в `hermes-workspace` одновременно запускали **генератор маршрутов на основе файлов TanStack Router**. Каждый из них записывал `src/routeTree.gen.ts`, обнаруживал изменение времени модификации (mtime) от другого процесса и перезапускал генератор. Это вызывало:

1. Постоянные предупреждения `routeTree.gen.ts was modified by another process during processing`.
2. Постоянные сигналы перезагрузки HMR, отправляемые всем браузерным клиентам.
3. Браузер заново выполнял цепочку заставки и запуска при каждой перезагрузке, не успевая стабилизироваться.

Это и был тот цикл загрузки, который видел Эрик. Слой авторизации/подключения никогда не был проблемой — страница просто уходила на перезагрузку из-за HMR каждые несколько сотен миллисекунд.

## Доказательства

- 34 039 случаев сообщения "modified by another process" в `/tmp/hermes3002.log` перед принудительным завершением (размер файла ~10 МБ).
- После завершения pids 40934, 8826, 8866 и запуска одного свежего vite на 3002:
  - время модификации `routeTree.gen.ts` **стабильно** при многократных проверках
  - размер лог-файла перестал расти
  - все эндпоинты состояния исправны (`/api/connection-status`, `/api/auth-check`, `/api/gateway-status`)

## Примененное исправление

1. Завершены три дублирующих процесса vite в hermes-workspace.
2. Запущен один свежий `pnpm dev` на порту 3002.
3. Файл `src/routes/api/send-stream-live-tools.ts` переименован в `src/routes/api/-send-stream-live-tools.ts`, чтобы префикс игнорирования TanStack корректно его пропускал. (Обновлены импорты в `send-stream.ts` и в файле тестов.) Это устраняет вторичное предупреждение "does not export a Route". **Это не было первопричиной**, но шум мешал при предыдущем расследовании.

Коммит: `72dd321e8` в ветке `main`.

## Как распознать это в будущем

Если на 3002 возникает цикл загрузки, **прежде чем** трогать код авторизации/запуска, проверьте:

```bash
ps aux | grep -i vite | grep -v grep
lsof -nP -iTCP:3002 -iTCP:3001 -iTCP:3003 -iTCP:3004 -iTCP:3005 -sTCP:LISTEN
```

Если к одному и тому же дереву исходников (`/Users/aurora/hermes-workspace`) привязано более одного vite, то это и есть баг. Серверы vite в рабочих деревьях (worktrees) с другими путями — это нормально.

Команда для обнаружения:

```bash
ps aux | grep -E "vite.*hermes-workspace/node_modules" | grep -v grep | grep -v worktree | wc -l
```

Должна возвращать `1` (или `0`, если dev-сервер не запущен). Любое число больше означает дубликаты.
